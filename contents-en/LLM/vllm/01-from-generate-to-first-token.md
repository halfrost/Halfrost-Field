## 1. Why the Request Path Is the Right First Read

vLLM is a large system: fused attention kernels, a block-based KV cache manager, a continuous-batching scheduler, tensor/pipeline/data parallelism, speculative decoding. It is tempting to begin with the kernel, because that is where much of vLLM's best-known work lives. That is the wrong first read. A PagedAttention kernel only ever sees a block table that some other layer built for it — read the kernel first and you are looking at the answer to a question you have not asked yet. Read one request end-to-end and every later subsystem stops being a free-floating optimization and becomes a named hop with an owner and an invariant.

The argument for *why the path matters at all* is a memory argument, and it is worth restating because it is the thing that makes the path more than plumbing. The launch post framed vLLM around a single bottleneck: LLM serving is memory-bound, dominated by KV cache memory, and PagedAttention exists to make that memory easy to allocate, share, and reuse ([launch](https://vllm.ai/blog/2023-06-20-vllm)). The SOSP paper formalizes the mechanism and reports a 2–4x throughput gain over prior systems such as FasterTransformer and Orca at comparable latency ([paper](https://arxiv.org/abs/2309.06180)). But notice where that memory management actually *happens*: not in the kernel, which merely dereferences a block table, but along the request path — where blocks are allocated on admission, shared via prefix caching, and freed on completion. The bottleneck that justifies vLLM's existence is only *legible* if you trace a request across those boundaries. That is the real reason to read the path first.

The driving question for this article is the one the series opens with:

> How does `LLM.generate()` become one model forward pass and one sampled token?

### The engine is four functions

Before any code, fix the vocabulary. The architecture overview describes the engine as four core functions, and this is the backbone of the entire series ([arch](https://docs.vllm.ai/en/stable/design/arch_overview/)):

> "Input Processing… Scheduling: 'Chooses which requests are processed in each step'… Model Execution: 'Manages the execution of the language model'… Output Processing: 'Processes the outputs generated by the model, decoding the token IDs'."

*Input processing* turns a raw prompt into an engine-shaped request — tokenization, multimodal preprocessing, parameter finalization ([Section 3](#3-input-processing-how-a-prompt-becomes-an-enginecorerequest)). *Scheduling* decides, each step, which requests run and how many tokens each gets ([Section 7](#7-scheduling-one-step-overview); deep dive article 05). *Model execution* runs the forward pass on the GPU and produces logits ([Section 8](#8-model-execution-from-executor-to-logits-overview)–[Section 9](#9-from-hidden-states-to-a-sampled-token-overview); articles 08–09). *Output processing* decodes token IDs back into user-visible text ([Section 10](#10-output-processing-enginecoreoutput-to-requestoutput); article 04). The later articles zoom in on these responsibilities and the boundaries between them.

### Those four functions are one repeated transaction

The four functions are not a pipeline you traverse once per request. Three of them — schedule, execute, process — are a single *transaction* that the engine runs over and over, once per "step", advancing the in-flight requests by a few tokens each turn, until each one finishes. This is not a metaphor. It is literally the body of one method.

[`vllm/v1/engine/core.py:479-508`](https://github.com/vllm-project/vllm/blob/6cf7b26bd4bff60bf378e1af14044280ac0d214c/vllm/v1/engine/core.py#L479-L508) (excerpt; [Section 6](#6-the-engine-step-schedule-execute-process-outputs) quotes the method in full):

```python

<a href='images/vllm-01-11-step-progression.svg' target='_blank'><img src='images/vllm-01-11-step-progression.svg' alt='vllm-01-11-step-progression'></a>

<p class='figure-caption'>Continuous batching in motion: each `EngineCore.step()` advances the requests granted token budget in that step, while requests enter and finish independently across steps — three of the four functions (schedule, execute, commit) repeat once per step while input processing runs once up front.</p>

The docstring, "Schedule, execute, and make output," is the architecture in four words. `schedule()` decides *how many tokens for which requests* without touching the GPU; `execute_model(..., non_block=True)` launches the forward pass and returns a `Future`, so the CPU can build the grammar bitmask *while the GPU computes*; `future.result()` is the synchronization point (and if execution deferred sampling, a separate `sample_tokens` call runs it); `_process_aborts_queue()` consumes cancellations that arrived during the frame before output reconciliation; and `update_from_output` folds sampled tokens into request progress. Section 6 returns to this method line by line. Articles 05, 08–10, and 04 cover the scheduler, execution, sampling, and lifecycle details respectively.

**Invariant.** The three generation surfaces discussed here — offline `LLM.generate`, `AsyncLLM.generate`, and the OpenAI-compatible server — ultimately submit work to the same EngineCore transaction and adapt its outputs. `generate()` itself does not schedule tokens, allocate KV blocks, or run kernels; it validates inputs, assigns request IDs, and drains the engine.

### The path is a chain of ownership boundaries

Because the engine is one small transaction wrapped in adapters, the request path reads as a *chain of ownership boundaries*, each protecting one contract. The public API owns validation, defaulting, and restoring input order. The input processor owns turning a prompt into an owned, fully-defaulted wire struct. The engine client owns a transport-agnostic surface. `EngineCore` owns the schedule/execute/commit transaction — but delegates all lifecycle state (which requests are waiting, running, or finished; who owns which blocks) to the scheduler. The scheduler owns token and memory decisions; the worker owns tensors and logits; the sampler owns token IDs; the output processor owns detokenization, stop strings, and the final `RequestOutput`. Each arrow between these owners is a later article. The value of naming them up front is that when a bug or a feature question arises — "who decides `max_tokens`?", "where does a stop string get detected?", "when are blocks freed?" — you already know which boundary to open.

### The path crosses processes, but the crossing is a deployment decision

The one complication article 01 must confront early is that these boundaries are not all in one process. V1 deliberately splits the four functions across processes so that CPU-bound work (tokenization, multimodal preprocessing, detokenization, streaming) overlaps with GPU-bound work ([v1](https://vllm.ai/blog/2025-01-27-v1-alpha-release), [arch](https://docs.vllm.ai/en/stable/design/arch_overview/)). The front-end process (`AsyncLLM`) handles HTTP, input processing, and streaming; the `EngineCore` process runs the scheduler, KV cache, and execution coordination; the workers run the forward pass. The concrete process count for a four-GPU tensor-parallel deployment is:

> 4-GPU TP=4 deployment = **1 API server + 1 engine core + 4 GPU workers = 6 processes** ([arch](https://docs.vllm.ai/en/stable/design/arch_overview/)); a DP Coordinator process is added under data parallelism.

The same configuration reaches all six processes coherently for a concrete reason: a single `VllmConfig` object is threaded through the entire stack: "By encapsulating all configurations in one object, we can easily pass the configuration object around and access the configuration we need" ([arch](https://docs.vllm.ai/en/stable/design/arch_overview/)).

The split does **not** leak into request semantics. `LLMEngine` holds exactly one handle to the engine, and the transport hides behind it.

[`vllm/v1/engine/llm_engine.py:104-111`](https://github.com/vllm-project/vllm/blob/6cf7b26bd4bff60bf378e1af14044280ac0d214c/vllm/v1/engine/llm_engine.py#L104-L111):

```python
        # EngineCore (gets EngineCoreRequests and gives EngineCoreOutputs)
        self.engine_core = EngineCoreClient.make_client(
            multiprocess_mode=multiprocess_mode,
            asyncio_mode=False,
            vllm_config=vllm_config,
            executor_class=executor_class,
            log_stats=self.log_stats,

```

The comment states the whole contract: the client "gets `EngineCoreRequest`s and gives `EngineCoreOutputs`." `make_client` picks the realization once, at construction, from `multiprocess_mode`: an in-process object that calls `EngineCore` directly, or a client that serializes requests over a ZMQ socket to a busy-looping `EngineCore` process. After this line, nothing on the request path knows or cares which one it got — `add_request` and `get_output` have identical signatures on both. (One correction worth carrying: even the "offline" `LLM` defaults to the multiprocess client, because `VLLM_ENABLE_V1_MULTIPROCESSING` defaults to true; the in-process path is the opt-out, not the default. Article 03 covers the transport fork in detail.)

## 2. LLM.generate(): The Offline Convenience Layer

Starting at the top of the map ([Section 1](#1-why-the-request-path-is-the-right-first-read)): `LLM.generate()` is the first ownership boundary in the request path, and it is deliberately boring. It does not schedule tokens, allocate KV blocks, or launch a single CUDA kernel. Its entire job is to validate the call, materialize concrete sampling parameters, mint request IDs, hand the work to the engine, and block until every result is back — then restore the caller's input order. Understanding precisely where that boundary sits, and which contracts it buys, is what lets the rest of this article treat everything below it as a black box that consumes `EngineCoreRequest`s and emits `RequestOutput`s.

The offline surface is really two files. `vllm/entrypoints/llm.py` owns validation and engine construction; `vllm/entrypoints/offline_utils.py` owns the render → add → drain spine. The `LLM` class itself is a façade assembled from mixins:

[`vllm/entrypoints/llm.py:66`](https://github.com/vllm-project/vllm/blob/6cf7b26bd4bff60bf378e1af14044280ac0d214c/vllm/entrypoints/llm.py#L66)

```python
class LLM(BeamSearchOfflineMixin, PoolingOfflineMixin, OfflineInferenceMixin):
```

The public methods (`generate`, `chat`, `enqueue`) live on `LLM` and do validation, default-filling, and orchestration — `enqueue` calls `_add_completion_requests()` to push requests into the engine queue and return their IDs ([`llm.py:487-530`](https://github.com/vllm-project/vllm/blob/6cf7b26bd4bff60bf378e1af14044280ac0d214c/vllm/entrypoints/llm.py#L487-L530)), while `generate` and `chat` delegate into the same render/add/drain path. Scheduling, memory management, and execution remain below the façade.

### Aligning the batch, then forcing FINAL_ONLY

`_run_completion` is two phases: enqueue everything, then block-drain the engine. Before a single request is admitted, `_add_completion_requests` normalizes the four positional inputs (prompts, params, LoRA, priority) into equal-length sequences. The scalar-broadcast-or-validate rule is the critical part:

[`vllm/entrypoints/offline_utils.py:247-256`](https://github.com/vllm-project/vllm/blob/6cf7b26bd4bff60bf378e1af14044280ac0d214c/vllm/entrypoints/offline_utils.py#L247-L256)

```python
        if isinstance(params, Sequence):
            if len(params) != num_requests:
                raise ValueError(
                    f"The lengths of prompts ({num_requests}) "
                    f"and params ({len(params)}) must be the same."
                )

            return params

        return [params] * num_requests
```

A single `SamplingParams` is broadcast to every prompt; a per-prompt list whose length does not equal the prompt count raises `ValueError` **before** any request is added. `_lora_request_to_seq` and `_priority_to_seq` follow the same shape. After this block, prompts, params, LoRA requests, and priorities are positionally aligned, so a length mismatch is rejected rather than pairing a prompt with a neighbor's settings. Prompts are then rendered *lazily*: prompt `i+1` can render while the engine is already working on prompt `i` in the default multiprocess mode. Passing a materialized list triggers a `warning_once` ([`offline_utils.py:504-512`](https://github.com/vllm-project/vllm/blob/6cf7b26bd4bff60bf378e1af14044280ac0d214c/vllm/entrypoints/offline_utils.py#L504-L512)).

Each aligned tuple reaches `_add_request`, which is where the offline path stamps its two signature decisions:

[`vllm/entrypoints/offline_utils.py:559-563`](https://github.com/vllm-project/vllm/blob/6cf7b26bd4bff60bf378e1af14044280ac0d214c/vllm/entrypoints/offline_utils.py#L559-L563)

```python
        if isinstance(params, SamplingParams):
            # We only care about the final output
            params.output_kind = RequestOutputKind.FINAL_ONLY

        request_id = str(next(self.request_counter))
```

First, **output kind is forced to `FINAL_ONLY`**, overwriting whatever the caller set. The enum makes the three modes explicit:

[`vllm/sampling_params.py:182-188`](https://github.com/vllm-project/vllm/blob/6cf7b26bd4bff60bf378e1af14044280ac0d214c/vllm/sampling_params.py#L182-L188)

```python
class RequestOutputKind(Enum):
    # Return entire output so far in every RequestOutput
    CUMULATIVE = 0
    # Return only deltas in each RequestOutput
    DELTA = 1
    # Do not return intermediate RequestOutput
    FINAL_ONLY = 2
```

<a href='images/vllm-01-13-output-kind-matrix.svg' target='_blank'><img src='images/vllm-01-13-output-kind-matrix.svg' alt='vllm-01-13-output-kind-matrix'></a>

<p class='figure-caption'>`RequestOutputKind` compared: `CUMULATIVE` re-sends the growing whole, `DELTA` emits only the new slice (with mailbox `aggregate=True`), and `FINAL_ONLY` emits exactly one output on finish — the mode `_add_request` forces for offline batch inference and that the drain loop's `if output.finished:` relies on.</p>

Offline batch inference does not stream; `FINAL_ONLY` tells the downstream `OutputProcessor` to emit one `RequestOutput` when the request finishes. This mutates the caller's `SamplingParams` object in place (pooling params are not `SamplingParams`, so they are left alone), and the drain loop's `if output.finished:` filter relies on that final-only behavior.

Second, **request IDs are monotonic decimal strings** drawn from the `Counter()` created in `__init__`: `"0"`, `"1"`, `"2"`, … in submission order. That ordering is the hook the final sort will exploit.

`_render_and_add_requests` also catches a mid-batch exception, aborts requests already added by that call, and then re-raises ([`offline_utils.py:545-548`](https://github.com/vllm-project/vllm/blob/6cf7b26bd4bff60bf378e1af14044280ac0d214c/vllm/entrypoints/offline_utils.py#L545-L548)). A failed render therefore does not leave the earlier members of the same batch running in the background.

### The blocking drain and input-order restoration

With requests admitted, `generate()` becomes a synchronous driver loop over the engine's own step:

[`vllm/entrypoints/offline_utils.py:594-599, 623-626`](https://github.com/vllm-project/vllm/blob/6cf7b26bd4bff60bf378e1af14044280ac0d214c/vllm/entrypoints/offline_utils.py#L594-L599)

<a href='images/vllm-01-14-order-restore.svg' target='_blank'><img src='images/vllm-01-14-order-restore.svg' alt='vllm-01-14-order-restore'></a>

<p class='figure-caption'>Why the terminal `sorted(...)` exists: continuous batching lets requests finish out of submission order (the scheduler freely interleaves prefill and decode), so monotonic decimal IDs minted from `Counter()` let the wrapper layer restore input order — keeping "preserve submission order" off the scheduler's hot path.</p>

The loop condition is the *engine's* bookkeeping — `has_unfinished_requests()` is backed by the output processor ([`llm_engine.py:188-195`](https://github.com/vllm-project/vllm/blob/6cf7b26bd4bff60bf378e1af14044280ac0d214c/vllm/v1/engine/llm_engine.py#L188-L195)), not a local counter — so the just-registered work is immediately visible and the loop keeps stepping until nothing is outstanding. Each `llm_engine.step()` is one full schedule → execute → process-outputs cycle (that wrapper is covered in this article's engine-step section, and the `EngineCore` loop it drives in article 04). The `assert isinstance(output, output_type)` is the runtime companion to the front-door gate: with `output_type=RequestOutput`, a pooling output would trip the assertion rather than being returned silently. Because params were forced `FINAL_ONLY`, `if output.finished:` collects each request exactly once. Finally, the results are `sorted` by integer request ID.

The terminal sort is why the monotonic IDs exist. Continuous batching allows a later request to finish before an earlier one, leaving the scheduler free to interleave prefill and decode across the running set. The comment at [`offline_utils.py:623-625`](https://github.com/vllm-project/vllm/blob/6cf7b26bd4bff60bf378e1af14044280ac0d214c/vllm/entrypoints/offline_utils.py#L623-L625) states the rationale. The API call remains blocking, and the wrapper restores input order after the engine has finished the batch. Article 05 explains why the scheduler benefits from that freedom.

### One user request may become n engine requests

The façade also holds the line on parallel sampling. When `SamplingParams.n > 1`, `LLMEngine.add_request` fans one logical request into `n` children, each with its own ID and child sampling params, but returns the *parent's* ID:

[`vllm/v1/engine/llm_engine.py:279-292`](https://github.com/vllm-project/vllm/blob/6cf7b26bd4bff60bf378e1af14044280ac0d214c/vllm/v1/engine/llm_engine.py#L279-L292)

```python
        # Fan out child requests (for n>1).
        parent_req = ParentRequest(request)
        for idx in range(n):
            request_id, child_params = parent_req.get_child_info(idx)
            child_request = request if idx == n - 1 else copy(request)
            child_request.request_id = request_id
            child_request.sampling_params = child_params

            # Make a new RequestState and queue.
            self.output_processor.add_request(
                child_request, prompt_text, parent_req, idx
            )
            # Add the request to EngineCore.
            self.engine_core.add_request(child_request)
```

Each child is registered on *both* sides — the output processor for detokenization and aggregation, and EngineCore for scheduling — while the `ParentRequest` merges them into one user-visible result. The offline `tqdm` accounting reflects this by advancing the bar by `n = len(output.outputs)`. Child sampling-parameter derivation belongs to article 10; two-sided registration to article 04.

## 3. Input Processing: How a Prompt Becomes an EngineCoreRequest

Between the public call and the scheduler sits `InputProcessor.process_inputs`. It takes heterogeneous input — a bare `str`, a `TextPrompt`, pre-tokenized IDs, precomputed embeddings, an encoder/decoder pair, or a multimodal bundle — and produces the wire-shaped `EngineCoreRequest` consumed downstream.

The important architectural fact is *where* this work runs. V1 deliberately moved tokenization, multimodal preprocessing, detokenization, and streaming **out of the hot execution loop** so CPU-bound preparation overlaps with GPU work ([V1 blog](https://vllm.ai/blog/2025-01-27-v1-alpha-release)). `process_inputs` is the front-end half of that split. By the time a request reaches `EngineCore`, tokenization is already done, params are already defaulted, and multimodal features are already flattened — the core loop never touches a tokenizer.

## 4. The Engine Boundary: In-Process vs the Multiprocess Client

The previous sections followed a prompt until it became an `EngineCoreRequest`. EngineCore may live in the same process as `LLM.generate()`, in a background process behind a ZMQ socket, or in one of several data-parallel engine processes. The frontend uses one client interface, with the transport fixed for that client's lifetime. This section follows that boundary and the data that crosses it.

### One handle, chosen once

`LLMEngine` (the synchronous engine used by the offline `LLM` path) owns exactly one channel to scheduling and execution. It is created in the constructor and typed as an abstract client, not as a concrete transport.

[`vllm/v1/engine/llm_engine.py:104-111`](https://github.com/vllm-project/vllm/blob/6cf7b26bd4bff60bf378e1af14044280ac0d214c/vllm/v1/engine/llm_engine.py#L104-L111)

```python

            asyncio_mode=False,

            executor_class=executor_class,

```

`self.engine_core` is the main route from `LLMEngine` to scheduling and execution. Operations such as `add_request`, `step`, `abort_request`, `sleep`, `add_lora`, and `collective_rpc` forward through that field. `asyncio_mode=False` selects the synchronous client family; the online `AsyncLLM` path passes `True`. `multiprocess_mode` comes from a classmethod constructor:

**Invariant.** The frontend depends on the `EngineCoreClient` interface rather than a concrete transport. `add_request`, `step`, and output processing do not branch on `multiprocess_mode`; client dispatch absorbs that choice.

### The factory is the fork point

The three transports live behind one abstract base class, and `make_client` is where the topology is chosen from two booleans.

### InprocClient: the boundary collapses to a method call

When the engine is in-process, the "client" is a thin shell that *owns* a real `EngineCore` and calls it directly — no serialization, no socket, no busy loop.

[`vllm/v1/engine/core_client.py:289-299`](https://github.com/vllm-project/vllm/blob/6cf7b26bd4bff60bf378e1af14044280ac0d214c/vllm/v1/engine/core_client.py#L289-L299)

        outputs, model_executed = self.engine_core.step_fn()
        self.engine_core.post_step(model_executed=model_executed)
        return outputs and outputs.get(0) or EngineCoreOutputs()

    def get_supported_tasks(self) -> tuple[SupportedTask, ...]:
        return self.engine_core.get_supported_tasks()

    def add_request(self, request: EngineCoreRequest) -> None:
        req, request_wave = self.engine_core.preprocess_add_request(request)
        self.engine_core.add_request(req, request_wave)
```

Two things fall out. First, `add_request` is exactly two inline calls on the caller's thread: `preprocess_add_request` turns the wire-shaped `EngineCoreRequest` into an internal `Request` (block-hashing it and initializing grammar), and `add_request` hands that `Request` to the scheduler. Second, `get_output` *drives* the engine one step: `step_fn` is the engine's own `step` (or `step_with_batch_queue` under pipeline parallelism), so in-process the front-end's `step()` call literally *is* the engine's step — there is no separate loop advancing anything. The `outputs.get(0)` picks engine index 0 because in-process there is exactly one engine. This is what the docstring means by "V0-style": admission and `step()` run on the caller's thread with no separate EngineCore busy loop. It describes the frontend-to-core boundary only — whether the executor and workers run concurrently still depends on the parallel configuration.

    def _send_input(self, request_type: EngineCoreRequestType, request: Any):
        self.ensure_alive()
        self.free_pending_messages()
        # (Identity, RequestType, SerializedRequest)
        msg = (self.core_engine, request_type.value, *self.encoder.encode(request))

        if len(msg) <= 3:
            # No auxiliary buffers => no tensor backing buffers in request.
            self.input_socket.send_multipart(msg, copy=False)
            return

        tracker = self.input_socket.send_multipart(msg, copy=False, track=True)
        self.add_pending_message(tracker, request)
```

<p class='figure-caption'>The `SyncMPClient` wire frame: an ordered multipart message `(engine_identity, one-byte type tag, *msgpack payload)` sent on a `zmq.ROUTER` to the engine's `zmq.DEALER`; when `len(msg) <= 3` it is a plain zero-copy `send_multipart`, otherwise out-of-band tensor buffers force `track=True` + `add_pending_message` to retain the buffer until ZMQ finishes sending — all fronted by `ensure_alive()`.</p>

The wire message is an ordered, self-describing multipart frame: `(engine_identity, one-byte type tag, *msgpack payload)`. The identity frame addresses a specific engine (essential once there are several under data parallelism); the type byte lets the receiver demux with no schema negotiation; the payload is `msgspec` msgpack. Three pieces of transport-specific bookkeeping appear here that `InprocClient` does not need: `ensure_alive()` raises `EngineDeadError` if the monitored engine process has died (so the boundary carries liveness, not just data); the `len(msg) <= 3` branch is zero-copy tensor safety — if the encoded request carried out-of-band tensor buffers (multimodal), the client must *retain a reference* via a `zmq.MessageTracker` until ZMQ finishes transmitting the backing memory, or it could be freed mid-send; and `add_request` is fire-and-forget — it returns after the send, and outputs come back asynchronously on a separate socket. `self.input_socket` is a `zmq.ROUTER` the client binds; the engine connects a `zmq.DEALER`, which is why the identity frame leads. The deep socket topology — how these sockets are wired, the DP coordinator, load-balanced routing, abort routing that must remember which engine holds each in-flight request — is the subject of **article 03**; here we only need the client contract.

### The wire vocabulary is one byte

The boundary's protocol is a tiny tagged union. Each request type is already represented as a single hex byte, so it requires no separate encoding step before being placed in the frame.

[`vllm/v1/engine/__init__.py:251-264`](https://github.com/vllm-project/vllm/blob/6cf7b26bd4bff60bf378e1af14044280ac0d214c/vllm/v1/engine/__init__.py#L251-L264)

```python
class EngineCoreRequestType(enum.Enum):
    """
    Request types defined as hex byte strings, so it can be sent over sockets
    without separate encoding step.
    """

    ADD = b"\x00"
    ABORT = b"\x01"
    START_DP_WAVE = b"\x02"
    UTILITY = b"\x03"
    # Sentinel used within EngineCoreProc.
    EXECUTOR_FAILED = b"\x04"
    # Sentinel to wake up input_queue.get() during shutdown.
    WAKEUP = b"\x05"
```

<a href='images/vllm-01-20-request-type-routing.svg' target='_blank'><img src='images/vllm-01-20-request-type-routing.svg' alt='vllm-01-20-request-type-routing'></a>

<p class='figure-caption'>`EngineCoreRequestType` routing: `ADD` / `ABORT` / `UTILITY` cross the socket client→engine (only `ADD` gets the typed `MsgpackDecoder(EngineCoreRequest, ...)`; the rest share an untyped decoder), `START_DP_WAVE` is coordinator→engine, and `EXECUTOR_FAILED` / `WAKEUP` are in-process sentinels that never travel the wire yet reuse the same one-demux dispatch path.</p>

`ADD`, `ABORT`, and `UTILITY` are the types that actually travel over the socket from the client; `START_DP_WAVE` is coordinator-to-engine; `EXECUTOR_FAILED` and `WAKEUP` are *in-process sentinels* pushed onto the engine's own input queue and never sent over the wire — they reuse the same dispatch path so the busy loop needs only one demux. Only `ADD` gets a dedicated *typed* decoder, `MsgpackDecoder(EngineCoreRequest, ...)`, because it deserializes into the typed `EngineCoreRequest` struct; every other type shares an untyped generic decoder ([`core.py:1494-1497`](https://github.com/vllm-project/vllm/blob/6cf7b26bd4bff60bf378e1af14044280ac0d214c/vllm/v1/engine/core.py#L1494-L1497)). (Both decoders are built with the same `oob_tensor_provider`, so out-of-band-tensor capability is *not* what distinguishes them, even though tensors most often ride along with `ADD` in the admission path; the protocol does not restrict tensor-bearing payloads to that type.) Return-valued operations (`get_supported_tasks`, `sleep`, `add_lora`, `collective_rpc`) all ride `UTILITY`, tagged with a `call_id` and correlated back to a `Future` so an inherently async socket round-trip presents a blocking API — in-process those same methods are plain Python returns.

### The symmetry: same lifecycle, relocated threads

The reason two transports can share one contract is that they perform the *same two operations in the same order* — the multiprocess path just splits them across a thread boundary. `InprocClient.add_request` calls `preprocess_add_request` then `scheduler`-bound `add_request` inline. The multiprocess engine does exactly the same pair, but `preprocess_add_request` runs on a dedicated input IO thread that decodes the frame, the resulting `(Request, wave)` tuple rides a thread-safe queue, and `add_request` runs on the busy-loop thread. The engine's own docstring names the intent:

[`vllm/v1/engine/core.py:855-860`](https://github.com/vllm-project/vllm/blob/6cf7b26bd4bff60bf378e1af14044280ac0d214c/vllm/v1/engine/core.py#L855-L860)

```python
    def preprocess_add_request(self, request: EngineCoreRequest) -> tuple[Request, int]:
        """Preprocess the request.

        This function could be directly used in input processing thread to allow
        request initialization running in parallel with Model forward
        """
```

This is the payoff of the process split: heavy request initialization (block hashing, grammar compilation) overlaps GPU work instead of blocking it. The in-process client runs the identical code with none of that pipelining, which is the price of running the engine on the caller's thread.

Both modes run `preprocess_add_request → scheduler.add_request`; multiprocess mode relocates the two halves onto different threads so initialization can overlap model execution. `get_output()` also keeps the same blocking signature: the in-process client steps EngineCore directly, while the multiprocess client waits on a queue fed by a socket-drain thread. Engine death and decode errors arrive through that queue as exception values, so the frontend's step loop uses the same method for data and failures.

This is the seam the rest of the request path hangs from. Upstream, the public API and input processor ([Section 2](#2-llmgenerate-the-offline-convenience-layer)–[Section 3](#3-input-processing-how-a-prompt-becomes-an-enginecorerequest) of this article) hand the client a fully-formed `EngineCoreRequest` and do not care where it goes. Downstream, `EngineCore.step()` — the `schedule → execute → commit` transaction covered next — runs identically whether it was invoked inline by `InprocClient.get_output()` or by the background process's busy loop. For the socket topology, DP routing, and coordinator internals that `SyncMPClient`/`AsyncMPClient` hide, see **article 03** (process architecture and ZMQ); for how parallelism multiplies engines and workers behind this same client, see **article 11**.

## 5. add_request: A Request Enters EngineCore

The transport is fixed at client construction and does not change the request representation. This section follows `add_request`: registration on two sides, parallel-sampling fan-out, and the point where the wire-shaped `EngineCoreRequest` becomes a scheduler-owned `Request`. Scheduler admission, KV blocks, grammar compilation, and child sampling continue in the later deep dives.

### Two registrars for one request

By the time control reaches the tail of `LLMEngine.add_request`, the raw prompt has already been through `input_processor.process_inputs` (article 01 [Section 3](#3-input-processing-how-a-prompt-becomes-an-enginecorerequest)) and is now an `EngineCoreRequest` with fully-defaulted, cloned `SamplingParams`. What remains is registration — and it is deliberately two-sided.

Source anchor: [`vllm/v1/engine/llm_engine.py:263-277`](https://github.com/vllm-project/vllm/blob/6cf7b26bd4bff60bf378e1af14044280ac0d214c/vllm/v1/engine/llm_engine.py#L263-L277)

```python
        self.input_processor.assign_request_id(request)

        req_id = request.request_id

        # Use cloned params that may have been updated in process_inputs()
        params = request.params

        n = params.n if isinstance(params, SamplingParams) else 1

        if n == 1:
            # Make a new RequestState and queue.
            self.output_processor.add_request(request, prompt_text, None, 0)
            # Add the request to EngineCore.
            self.engine_core.add_request(request)
            return req_id
```

`assign_request_id` stamps the internal id (external id plus 8 random characters — article 01 [Section 3](#3-input-processing-how-a-prompt-becomes-an-enginecorerequest)); `req_id` is captured *here*, before any fan-out, so it is the id the caller gets back. The number of samples `n` is read from the sampling params (pooling requests are always `n == 1`). Then the fast path does exactly two things, in this order:

1. `output_processor.add_request(request, prompt_text, None, 0)` — creates the `RequestState` that will detokenize, aggregate, and (online) hold the caller's output mailbox. This is the *return-path* registration.
2. `engine_core.add_request(request)` — hands the request to the client, which submits it for scheduling. This is the *forward-path* registration.

The ordering is not incidental. The output side is registered **before** the request is submitted for execution, so no `EngineCoreOutput` can ever arrive for a request the `OutputProcessor` has not yet heard of. This closes an output-before-registration race that would otherwise drop the first token of a fast-completing request. The offline drain loop sees the registration immediately for the same reason:

Source anchor: [`vllm/v1/engine/llm_engine.py:188-195`](https://github.com/vllm-project/vllm/blob/6cf7b26bd4bff60bf378e1af14044280ac0d214c/vllm/v1/engine/llm_engine.py#L188-L195)

```python
    def get_num_unfinished_requests(self) -> int:
        return self.output_processor.get_num_unfinished_requests()

    def has_unfinished_requests(self) -> bool:
        has_unfinished = self.output_processor.has_unfinished_requests()
        if self.dp_group is None:
            return has_unfinished or self.engine_core.dp_engines_running()
        return self.has_unfinished_requests_dp(has_unfinished)
```

Both "is there work left?" predicates are backed by the `output_processor`, not by the engine core. So the moment step (1) above returns, `LLM._run_engine`'s `while has_unfinished_requests():` loop (article 01 [Section 2](#2-llmgenerate-the-offline-convenience-layer)) will keep driving `step()` until this request finishes — even before the engine process has acknowledged it. The output processor is the authority on liveness; the engine core is where the work happens.

### One user request, n engine requests

The interesting case is `n > 1` — parallel sampling, where the user asks for several completions of one prompt. vLLM does not teach the scheduler about "n samples." It fans the single request into `n` independent engine requests and reassembles them on the way out.

Source anchor: [`vllm/v1/engine/llm_engine.py:279-294`](https://github.com/vllm-project/vllm/blob/6cf7b26bd4bff60bf378e1af14044280ac0d214c/vllm/v1/engine/llm_engine.py#L279-L294)

```python
        # Fan out child requests (for n>1).
        parent_req = ParentRequest(request)
        for idx in range(n):
            request_id, child_params = parent_req.get_child_info(idx)
            child_request = request if idx == n - 1 else copy(request)
            child_request.request_id = request_id
            child_request.sampling_params = child_params

            # Make a new RequestState and queue.
            self.output_processor.add_request(
                child_request, prompt_text, parent_req, idx
            )
            # Add the request to EngineCore.
            self.engine_core.add_request(child_request)

        return req_id
```

A `ParentRequest` wraps the original. For each child index it mints a child id and child sampling params:

Source anchor: [`vllm/v1/engine/parallel_sampling.py:83-94`](https://github.com/vllm-project/vllm/blob/6cf7b26bd4bff60bf378e1af14044280ac0d214c/vllm/v1/engine/parallel_sampling.py#L83-L94)

```python
    def get_child_info(self, index: int) -> tuple[str, SamplingParams]:
        """Get child request ID and sampling params.

        Args:
          index: index within `n` child requests.

        Returns:
          (request ID, sampling_params) tuple
        """
        child_req_id = f"{index}_{self.request_id}"
        self.child_requests.add(child_req_id)
        return child_req_id, self._get_child_sampling_params(index)
```

Each child id is the parent id prefixed with its index (`0_<parent>`, `1_<parent>`, …); the child sampling params are derived per-child (the seed handling and the `n → 1` collapse for each child belong to sampling — article 01 [Section 9](#9-from-hidden-states-to-a-sampled-token-overview) / deep article 10). The last child (`idx == n - 1`) *reuses the original request object* rather than copying it, an allocation optimization; the earlier children are `copy(request)`. Every child is registered on both sides, just as the `n == 1` fast path did — but each `output_processor.add_request` now passes the shared `parent_req` and the child's `idx`, which is how the `OutputProcessor` later merges the `n` streams back into one.

**Invariant.** Parallel sampling is represented as `n` engine requests beneath one user-visible request. The scheduler and workers handle flat child requests; `OutputProcessor` owns their aggregation, cancellation, and output ordering.

### Crossing the boundary: EngineCoreRequest becomes Request

`self.engine_core.add_request(child_request)` looks like a single call, but a type change happens inside it that is the real entry into engine ownership. The object the frontend passes is an `EngineCoreRequest` — the GC-free msgspec wire struct. The object the engine's scheduler receives is a `vllm.v1.request.Request`. The conversion sits in `preprocess_add_request`:

Source anchor: [`vllm/v1/engine/core.py:855-877`](https://github.com/vllm-project/vllm/blob/6cf7b26bd4bff60bf378e1af14044280ac0d214c/vllm/v1/engine/core.py#L855-L877)

```python
    def preprocess_add_request(self, request: EngineCoreRequest) -> tuple[Request, int]:
        """Preprocess the request.

        This function could be directly used in input processing thread to allow
        request initialization running in parallel with Model forward
        """

        if req.use_structured_output:
            ...

```

`Request.from_engine_core_request` materializes the internal request and runs the block hasher over the prompt tokens (seeding prefix-cache lookup — article 01 [Section 7](#7-scheduling-one-step-overview) / deep article 07). For structured-output requests, `grammar_init` kicks off grammar compilation. The docstring states the design intent verbatim: this "could be directly used in input processing thread to allow request initialization running in parallel with Model forward." That is precisely where the two transports diverge, as the engine-boundary section established: `InprocClient.add_request` runs `preprocess_add_request` then `add_request` inline on the caller's thread, while `EngineCoreProc` runs `preprocess_add_request` on its input IO thread and lets the busy loop call `add_request` — same two operations, same order, relocated to overlap block-hashing and grammar compile with the GPU forward pass (article 01 [Section 4](#4-the-engine-boundary-in-process-vs-the-multiprocess-client); deep article 03 for the ZMQ threading). Either way, what lands on the engine's scheduling side is a `Request`, never the wire struct.

### Engine-side admission

The engine-side `add_request` — the method the busy loop (or `InprocClient`) finally calls — is short, and its shortness is the point.

Source anchor: [`vllm/v1/engine/core.py:372-407`](https://github.com/vllm-project/vllm/blob/6cf7b26bd4bff60bf378e1af14044280ac0d214c/vllm/v1/engine/core.py#L372-L407)

```python
    def add_request(self, request: Request, request_wave: int = 0):
        """Add request to the scheduler.

        `request_wave`: indicate which wave of requests this is expected to
        belong to in DP case
        """
        # Validate the request_id type.
        if not isinstance(request.request_id, str):
            raise TypeError(
                f"request_id must be a string, got {type(request.request_id)}"
            )

        if pooling_params := request.pooling_params:
            supported_pooling_tasks = [
                task for task in self.get_supported_tasks() if task in POOLING_TASKS
            ]

            if pooling_params.task not in supported_pooling_tasks:
                raise ValueError(
                    f"Unsupported task: {pooling_params.task!r} "
                    f"Supported tasks: {supported_pooling_tasks}"
                )

        if request.kv_transfer_params is not None and (
            not self.scheduler.get_kv_connector()
        ):
            logger.warning(
                "Got kv_transfer_params, but no KVConnector found. "
                "Disabling KVTransfer for this request."
            )

        self.scheduler.add_request(request)
        if request.abort_immediately:
            # Immediately abort so the connector's request_finished hook runs
            # to free any pre-admission KV-transfer resources.
            self.abort_requests([request.request_id])
```

<a href='images/vllm-01-21-engine-admission.svg' target='_blank'><img src='images/vllm-01-21-engine-admission.svg' alt='vllm-01-21-engine-admission'></a>

<p class='figure-caption'>Engine-side `add_request`: three guards (request-id `str`, pooling-task membership, KV-transfer-without-connector warning), one handoff to `self.scheduler.add_request(request)`, and the `abort_immediately` edge — after which the scheduler alone owns the WAITING→RUNNING→FINISHED state machine, and abort is just `finish_requests(ids, FINISHED_ABORTED)`.</p>

Three guards, one handoff, one edge case:

- **Type guard** — the request id must be a `str`; this is asserted again here even though the frontend already checked, because in multiprocess mode the object arrived over a socket and the engine trusts nothing.
- **Pooling-task guard** — if the request carries `pooling_params`, its task must be in the model's supported pooling tasks, else `ValueError`. Generation requests skip this.
- **KV-transfer warning** — a request with `kv_transfer_params` but no configured `KVConnector` is not rejected; the transfer is silently disabled with a warning (disaggregated prefill territory, deep article 11).
- **The handoff** — `self.scheduler.add_request(request)`. That single line is the crossing into scheduler ownership. From here the request lives in the waiting queue, its blocks are managed by the KV cache manager, and its status machine (WAITING → RUNNING → FINISHED) is the scheduler's alone. `EngineCore.add_request` does not touch a counter, allocate a block, or set a status — it validates and delegates. What `scheduler.add_request` does with the request — enqueue it, register it with the KV connector, record a QUEUED event — is article 01 [Section 7](#7-scheduling-one-step-overview) / deep article 05.
- **The edge case** — `abort_immediately`. A request can be admitted and instantly aborted in the same call so that the KV connector's `request_finished` hook runs to release resources that were reserved before admission. This is not an error path bolted on; it is admission and teardown expressed through the *same* two primitives.

### Abort is just finish

That last point generalizes into the ownership contract of the engine core as a whole. Abort does not have its own machinery:

Source anchor: [`vllm/v1/engine/core.py:409-415`](https://github.com/vllm-project/vllm/blob/6cf7b26bd4bff60bf378e1af14044280ac0d214c/vllm/v1/engine/core.py#L409-L415)

```python
    def abort_requests(self, request_ids: list[str]):
        """Abort requests from the scheduler."""

        # TODO: The scheduler doesn't really need to know the
        # specific finish reason, TBD whether we propagate that
        # (i.e. client-aborted vs stop criteria met).
        self.scheduler.finish_requests(request_ids, RequestStatus.FINISHED_ABORTED)
```

Aborting a request is `scheduler.finish_requests(ids, FINISHED_ABORTED)` — the exact same call the scheduler uses when a request finishes normally, differing only in the terminal status. There is no separate "cancel" pathway that frees blocks a different way, which is what makes abort race-safe: whether a request stops because it hit EOS, hit a stop string (promoted by the `OutputProcessor`, article 01 [Section 10](#10-output-processing-enginecoreoutput-to-requestoutput)), or was cancelled by a disconnecting client, it exits through one state machine that owns all block-freeing (deep article 04 covers `finish_requests` / `_free_request` and the deferred, GPU-step-fenced frees). `EngineCore` never frees a block or advances a counter directly; every lifecycle mutation is behind `self.scheduler`.

### The async mirror

The online path registers the same two sides in the same order, with one addition — a per-request output mailbox created *before* submission.

Source anchor: [`vllm/v1/engine/async_llm.py:400-412`](https://github.com/vllm-project/vllm/blob/6cf7b26bd4bff60bf378e1af14044280ac0d214c/vllm/v1/engine/async_llm.py#L400-L412)

```python
    async def _add_request(
        self,
        request: EngineCoreRequest,
        prompt: str | None,
        parent_req: ParentRequest | None,
        index: int,
        queue: RequestOutputCollector,
    ):
        # Add the request to OutputProcessor (this process).
        self.output_processor.add_request(request, prompt, parent_req, index, queue)

        # Add the EngineCoreRequest to EngineCore (separate process).
        await self.engine_core.add_request_async(request)
```

The caller (`AsyncLLM.add_request`) first builds a `RequestOutputCollector` — the single-slot mailbox that `generate()` will later drain (article 01 [Section 11](#11-streaming-the-async-generator-and-why-the-first-token-is-special)) — and passes it here as `queue`. `_add_request` registers the `RequestState` (now holding that mailbox) with the `OutputProcessor`, *then* `await`s the submission to the engine core. The comments make the process split explicit: output processing is "this process," the engine core is a "separate process." Because both steps sit in one un-interleaved `await` chain and registration precedes the submit await, the mailbox provably exists before any output for the request can be produced. The sync `LLMEngine.add_request` gets the same guarantee for free — its two registrations are plain sequential calls on one thread. Register-before-submit is the same contract on both paths; asyncio just makes the ordering something you have to be deliberate about.

## 6. The Engine Step: Schedule, Execute, Process Outputs

Everything up to this point has adapted or transported the request. `EngineCore.step()` is where the runtime schedules work, launches execution, handles sampling and aborts, and reconciles output. The "Inside vLLM" walkthrough summarizes the same loop as selecting requests, running the model, sampling tokens, updating state, and freeing finished resources ([Inside vLLM](https://vllm.ai/blog/2025-09-05-anatomy-of-vllm)).

<a href='images/vllm-01-02-engine-step.svg' target='_blank'><img src='images/vllm-01-02-engine-step.svg' alt='vllm-01-02-engine-step'></a>

### The four jobs of `EngineCore.step()`

One step asks the scheduler what to run, launches the forward pass without blocking, prepares the structured-output bitmask concurrently, samples if execution deferred it, drains queued aborts, and then reconciles sampled tokens with request state.

Source: [`vllm/v1/engine/core.py:479-508`](https://github.com/vllm-project/vllm/blob/6cf7b26bd4bff60bf378e1af14044280ac0d214c/vllm/v1/engine/core.py#L479-L508)

```python
    def step(self) -> tuple[dict[int, EngineCoreOutputs], bool]:
        """Schedule, execute, and make output.

        Returns tuple of outputs and a flag indicating whether the model
        was executed.
        """

        # Check for any requests remaining in the scheduler - unfinished,
        # or finished and not yet removed from the batch.
        if not self.scheduler.has_requests():
            return {}, False
        scheduler_output = self.scheduler.schedule(self._should_throttle_prefills())
        future = self.model_executor.execute_model(scheduler_output, non_block=True)
        grammar_output = self.scheduler.get_grammar_bitmask(scheduler_output)
        with (
            self.log_error_detail(scheduler_output),
            self.log_iteration_details(scheduler_output),
        ):
            model_output = future.result()
            if model_output is None:
                model_output = self.model_executor.sample_tokens(grammar_output)

        # Before processing the model output, process any aborts that happened
        # during the model execution.
        self._process_aborts_queue()
        engine_core_outputs = self.scheduler.update_from_output(
            scheduler_output, model_output
        )

        return engine_core_outputs, scheduler_output.total_num_scheduled_tokens > 0
```


**Job 0 — the guard ([`core.py:488-489`](https://github.com/vllm-project/vllm/blob/6cf7b26bd4bff60bf378e1af14044280ac0d214c/vllm/v1/engine/core.py#L488-L489)).** `if not self.scheduler.has_requests(): return {}, False`. No model runs when the scheduler is empty. The comment above it includes requests that are finished but not yet removed from the batch, so `has_requests()` is wider than "requests still generating." It remains true while completion or connector work still needs to drain. Article 04 follows that tail of the lifecycle.

**Job 1 — schedule ([`core.py:490`](https://github.com/vllm-project/vllm/blob/6cf7b26bd4bff60bf378e1af14044280ac0d214c/vllm/v1/engine/core.py#L490)).** `scheduler.schedule(self._should_throttle_prefills())` returns a `SchedulerOutput` and runs *no model code*. It decides which requests run, how many new tokens each gets, which KV blocks are allocated or reused, and which requests are preempted. The resulting request and token-allocation state lets the model runner update its persistent batch and build the attention and sampling state required for execution. `_should_throttle_prefills()` is `False` in the base class ([`core.py:474-477`](https://github.com/vllm-project/vllm/blob/6cf7b26bd4bff60bf378e1af14044280ac0d214c/vllm/v1/engine/core.py#L474-L477)) and only overridden by the data-parallel engine for prefill balancing. What "how many tokens for which requests" means — continuous batching, chunked prefill, prefix-cache reuse — is the entire subject of article 05; article 01 states only that this one call is the whole scheduling decision for the step.

**Job 2 — execute, overlap, sample ([`core.py:491-499`](https://github.com/vllm-project/vllm/blob/6cf7b26bd4bff60bf378e1af14044280ac0d214c/vllm/v1/engine/core.py#L491-L499)).** This is the connective-tissue trick. `execute_model(scheduler_output, non_block=True)` returns a `Future` immediately instead of blocking on the GPU. That gap is filled by `get_grammar_bitmask(scheduler_output)`, which builds the structured-output token mask on the CPU *while the forward pass runs*. `future.result()` is the synchronization point. The `model_output is None` sentinel means the executor deferred sampling to a separate call, `sample_tokens(grammar_output)`, so a structured-output mask is applied against *this* step's logits at sampling time. The executor→worker→model-runner→logits chain behind `execute_model` is article 08/09; the logits→token-ids step behind `sample_tokens` is article 10. Two facts matter for the end-to-end path: the grammar bitmask comes from the *scheduler*, not the HTTP layer, and CPU mask preparation overlaps GPU execution, so its cost may be partly or fully hidden depending on their relative durations.

**The abort window ([`core.py:501-503`](https://github.com/vllm-project/vllm/blob/6cf7b26bd4bff60bf378e1af14044280ac0d214c/vllm/v1/engine/core.py#L501-L503)).** `self._process_aborts_queue()` runs *after* `future.result()` — the GPU frame is done — but *before* `update_from_output`. This ordering is the whole point: an external cancellation that arrived while the GPU was busy is consumed before the sampled tokens are folded into request state, so a request cancelled mid-frame is never advanced. The drain helper ([`core.py:634-642`](https://github.com/vllm-project/vllm/blob/6cf7b26bd4bff60bf378e1af14044280ac0d214c/vllm/v1/engine/core.py#L634-L642)) batches the entire queue into one `abort_requests` call; because aborting an already-finished request is idempotent, aborts can safely be dual-queued (see article 04).

**Job 3 — commit ([`core.py:504-506`](https://github.com/vllm-project/vllm/blob/6cf7b26bd4bff60bf378e1af14044280ac0d214c/vllm/v1/engine/core.py#L504-L506)).** `scheduler.update_from_output(scheduler_output, model_output)` is where sampled tokens become request progress: output ids are appended, finish reasons are set, and cleanup begins for finished requests. Their KV blocks are freed immediately when safe or placed in `deferred_frees` while an in-flight GPU step may still reference them. The method returns a `dict[client_index → EngineCoreOutputs]` keyed by owning client, and the second tuple element — `scheduler_output.total_num_scheduled_tokens > 0` — is the `model_executed` flag the loop wrapper consumes.

**Invariant.** Scheduling may reserve blocks and update planned counters before execution, but sampled tokens, terminal status, and the corresponding completion work are reconciled through `update_from_output` after model output arrives. Aborts queued during the GPU frame are drained before that reconciliation.

### Why the caller does not see this method

`EngineCore.step()` returns `EngineCoreOutputs` — token ids, finish flags, per-client routing — not user-facing `RequestOutput`s. The offline caller never touches it. What `LLM._run_engine`'s drain loop calls is `LLMEngine.step()`, a synchronous wrapper in a different file and a different process from the core loop.

Source: [`vllm/v1/engine/llm_engine.py:296-334`](https://github.com/vllm-project/vllm/blob/6cf7b26bd4bff60bf378e1af14044280ac0d214c/vllm/v1/engine/llm_engine.py#L296-L334)

        if self.should_execute_dummy_batch:
            self.should_execute_dummy_batch = False
            self.engine_core.execute_dummy_batch()
            return []

        # 1) Get EngineCoreOutput from the EngineCore.
        with record_function_or_nullcontext("llm_engine step: get_output"):
            outputs = self.engine_core.get_output()

        # 2) Process EngineCoreOutputs.
        with record_function_or_nullcontext("llm_engine step: process_outputs"):
            iteration_stats = IterationStats() if self.log_stats else None

                outputs.outputs,
                engine_core_timestamp=outputs.timestamp,
                iteration_stats=iteration_stats,
            )
            self.output_processor.update_scheduler_stats(outputs.scheduler_stats)

        # 3) Abort any reqs that finished due to stop strings.
        with record_function_or_nullcontext("llm_engine step: abort_requests"):
            self.engine_core.abort_requests(processed_outputs.reqs_to_abort)

        # 4) Record stats
        with record_function_or_nullcontext("llm_engine step: record_stats"):
            if (
                self.logger_manager is not None
                and outputs.scheduler_stats is not None
                and len(outputs.outputs) > 0
            ):
                self.logger_manager.record(
                    scheduler_stats=outputs.scheduler_stats,
                    iteration_stats=iteration_stats,

This is why `EngineCoreOutput.finished` and the processor's finish reason can legitimately diverge for one step.
- **The dummy-batch fast path (lines 297-300)** exists for data-parallel lockstep: a rank with no work still must execute an empty batch so its peers can proceed.


### The same step, wrapped for a background process

In the default multiprocess deployment ([Section 4](#4-the-engine-boundary-in-process-vs-the-multiprocess-client)), `EngineCore.step()` does not run on the caller thread at all — it runs inside a perpetual loop in the engine process:

Source: [`vllm/v1/engine/core.py:1259-1267`](https://github.com/vllm-project/vllm/blob/6cf7b26bd4bff60bf378e1af14044280ac0d214c/vllm/v1/engine/core.py#L1259-L1267)

    def run_busy_loop(self):
        """Core busy loop of the EngineCore."""
        while self._handle_shutdown():
            # 1) Poll the input queue until there is work to do.
            self._process_input_queue()
            # 2) Step the engine core and return the outputs.
            self._process_engine_step()

```

Each turn absorbs client input, then takes exactly one step via `_process_engine_step` ([`core.py:1300-1317`](https://github.com/vllm-project/vllm/blob/6cf7b26bd4bff60bf378e1af14044280ac0d214c/vllm/v1/engine/core.py#L1300-L1317)), which calls `self.step_fn()`, pushes each `(client_index, EngineCoreOutputs)` pair onto the output queue for the IO thread to serialize, runs the spec-decode `post_step` hook, and — if the step scheduled nothing runnable but requests remain — sleeps 1 ms to yield the GIL so background KV-transfer threads make progress. `step_fn` itself is bound once at construction to either `step` or the pipeline-parallel `step_with_batch_queue` ([`core.py:221-224`](https://github.com/vllm-project/vllm/blob/6cf7b26bd4bff60bf378e1af14044280ac0d214c/vllm/v1/engine/core.py#L221-L224)), so the loop never branches on parallelism per iteration. The busy-loop mechanics, shutdown draining, and finished-request bookkeeping are the deep subject of article 04; article 01's claim is only structural: **the sync offline `LLMEngine.step()` and the async busy loop are the same `EngineCore.step()` transaction wrapped two different ways** — driven inline by `InprocClient`, or by the EngineCore process's busy loop with `SyncMPClient` (the default) blocking the offline caller on an output queue and `AsyncMPClient` dispatching outputs asynchronously. The transaction is contract; only its driver changes.

## 7. Scheduling One Step (Overview)

We just saw `EngineCore.step()` open with a single line — `scheduler.schedule(...)` — and then hand a `SchedulerOutput` to the executor. This section stays at the overview altitude the rest of this article lives at: it explains *what the scheduler decides*, the *one idea* that lets prefill, decode, chunked prefill, prefix caching, and speculative decoding all share a single loop, and *what crosses the scheduler boundary in each direction*. The internals — the queues, the token-budget accounting, `allocate_slots`, preemption — belong to Article 05 (Scheduler: Continuous Batching and Chunked Prefill). Here we only wire the box into the request path.

### The scheduler decides tokens, not tensors

The scheduler is a pure planner. It runs on the CPU, inside the EngineCore process, and it never touches the GPU. One call answers a single question: *of the requests I am tracking, which ones run this step, and how many tokens does each one advance?* The answer is a `SchedulerOutput` — a plan. Executing that plan is a separate, later phase of the same `step()`.

- The guard at `:488-489` means the model is never run without a plan; if the scheduler holds nothing (not even a finished-but-unflushed request — see Article 04), the step short-circuits and reports `model_executed = False`.
- `schedule()` returns a `SchedulerOutput` and returns immediately — no forward pass, no logits. Everything after this line in `step()` (the `execute_model` future, the grammar bitmask, `sample_tokens`, `update_from_output`) consumes that plan; none of it is the plan.
- `_should_throttle_prefills()` is `False` in the base class and only overridden by the data-parallel engine core to defer new prefills for cross-replica balancing (Article 11, Distributed Inference and Parallelism). For the single-engine path you are tracing, the argument is a constant `False`.


### One idea: no prefill phase, no decode phase

The V1 blog frames the central simplification: rather than treating prefill and decode as distinct scheduler phases, V1 treats all tokens uniformly, and a scheduling decision becomes conceptually a dictionary, `{request_id: num_tokens}`. Chunked prefill uses the same model: a long prompt receives part of the step's token budget and continues later ([V1 blog](https://vllm.ai/blog/2025-01-27-v1-alpha-release)). The source states the same idea at the top of `schedule()`.

Source anchor: [`vllm/v1/core/sched/scheduler.py:396-407`](https://github.com/vllm-project/vllm/blob/6cf7b26bd4bff60bf378e1af14044280ac0d214c/vllm/v1/core/sched/scheduler.py#L396-L407)

```python
    def schedule(self, throttle_prefills: bool = False) -> SchedulerOutput:
        self.current_step += 1
        # NOTE(woosuk) on the scheduling algorithm:
        # There's no "decoding phase" nor "prefill phase" in the scheduler.
        # Each request just has the num_computed_tokens and
        # num_tokens_with_spec. num_tokens_with_spec =
        # len(prompt_token_ids) + len(output_token_ids) + len(spec_token_ids).
        # At each step, the scheduler tries to assign tokens to the requests
        # so that each request's num_computed_tokens can catch up its
        # num_tokens_with_spec. This is general enough to cover
        # chunked prefills, prefix caching, speculative decoding,
        # and the "jump decoding" optimization in the future.
```

<a href='images/vllm-01-23-counter-model.svg' target='_blank'><img src='images/vllm-01-23-counter-model.svg' alt='vllm-01-23-counter-model'></a>

<p class='figure-caption'>The unified counter model: scheduling closes the gap between `num_computed_tokens` (KV filled) and `num_tokens_with_spec` (prompt + output + draft) by granting some tokens this step — a fresh prompt, chunked prefill, a one-token decode, a prefix-cache jump, and speculative draft tokens are all the same gap-closing operation, so the request path branches only on "how many tokens."</p>


- Every request carries two counters: `num_computed_tokens` (how far the KV cache has been filled) and `num_tokens_with_spec` (prompt + generated + any speculative draft tokens). Scheduling is the act of closing that gap by some number of tokens this step.
- A fresh prompt starts with `num_computed_tokens = 0` and a large gap, so it looks like a big token demand; a decoding request has a gap of one (plus draft tokens under speculation). Both are the same operation — "advance `num_computed_tokens`" — differing only in the number granted.
- Because the granted amount can be *less* than the gap, chunked prefill needs no special code path: it is just a prompt that does not receive its whole gap in one step.
- Prefix caching and speculative decoding are named in the same comment as instances of the same generality: prefix caching lowers a request's effective gap by reusing already-computed blocks (Article 07, Automatic Prefix Caching); speculation raises `num_tokens_with_spec` with draft tokens to be verified (Article 12, Speculative Decoding).


### Priority: advance running requests before admitting waiting ones

Given a fixed per-step token budget, the scheduler advances requests already in the `running` queue before pulling waiting prefills into the remaining budget — which usually favors decode latency, though a running request may still be finishing a chunked prefill ([Inside vLLM](https://vllm.ai/blog/2025-09-05-anatomy-of-vllm)). The loop order in the source makes this literal.

Source anchor: [`vllm/v1/core/sched/scheduler.py:440-442`](https://github.com/vllm-project/vllm/blob/6cf7b26bd4bff60bf378e1af14044280ac0d214c/vllm/v1/core/sched/scheduler.py#L440-L442)

```python

        req_index = 0
        while req_index < len(self.running) and token_budget > 0:
```


- `running` is drained first, against `token_budget`; a second pass (not shown — Article 05) admits waiting requests with whatever budget remains.
- This is precisely why continuous batching and chunked prefill coexist in *one* loop rather than alternating phases: a step can carry a batch of in-flight decodes *and* a slice of a new prompt's prefill together, because both are just token grants against the same budget.

In this loop, running requests are considered before waiting prefills. A large prompt uses remaining token budget and may be chunked across steps. Article 05 covers the queue mechanics, preemption, and budget math behind that policy.

The `SchedulerOutput` is the request path's payload from CPU planning to GPU execution: it names the new and cached requests, the per-request token counts, KV block assignments, and spec-decode tokens (the model runner builds the attention metadata from this state). - `finished_req_ids` (a `set`) rides the *next* `SchedulerOutput` so the worker can prune the finished request from its persistent batch state (recall the V1 differential-batch design, where the worker caches request state across steps).

Finishing a request produces bookkeeping for two consumers on different clocks: the client can observe completion in the current output, while worker-side state is removed through subsequent scheduler feedback.

### Freeing blocks, fenced behind the GPU

The scheduler owns KV block ownership, and freeing blocks is where the "plan now, execute later" split becomes a correctness hazard: a request may finish while a forward pass it was part of is still writing its KV. So block frees can be *deferred* and are fenced behind the in-flight step.

Source anchor: [`vllm/v1/core/sched/scheduler.py:2138-2151`](https://github.com/vllm-project/vllm/blob/6cf7b26bd4bff60bf378e1af14044280ac0d214c/vllm/v1/core/sched/scheduler.py#L2138-L2151)

```python
    def _free_request_blocks(self, request: Request):
        """Free the request's KV blocks, deferring the return to the block
        pool when an in-flight GPU step may still write them.
        """
        if not self.defer_block_free or (
            # Last scheduled step already processed: no in-flight write remains
            # (always the case for a normal finish), so free now.
            request.last_sched_seq <= self.processed_step_seq
        ):
            self.kv_cache_manager.free(request)
            return
        blocks = self.kv_cache_manager.pop_blocks_for_free(request)
        if blocks:
            self.deferred_frees.append((self.sched_step_seq, blocks))
```

<p class='figure-caption'>Finishing fans out on two clocks: `finished_req_ids` rides the *next* `SchedulerOutput` so the worker prunes the request from its persistent batch, while `finished_req_ids_dict` rides *this* step's `EngineCoreOutputs.finished_requests` so the client retires its `RequestState` — and `_free_request_blocks` frees now or parks blocks in `deferred_frees` until any in-flight GPU write completes.</p>


- The common case (`not self.defer_block_free`, or the request's last scheduled step already processed) frees immediately — a normal end-of-generation finish always hits this branch.
- When async scheduling could have a GPU frame still writing this request's blocks (`last_sched_seq > processed_step_seq`), the blocks are parked in `deferred_frees` keyed by the step sequence and only returned once that step completes.

**Invariant.** A cache block is not returned to the pool while an enqueued GPU write may still reference it. Articles 06 and 07 cover the allocator, deferred frees, and prefix-cache bookkeeping behind that rule.

        self, grammar_output: GrammarOutput | None, non_block: bool = False
    ) -> ModelRunnerOutput | Future[ModelRunnerOutput]:
        output = self.collective_rpc(  # type: ignore[call-overload]
            "sample_tokens", args=(grammar_output,), non_block=non_block
        )
        return output[0]
```


- Both public executor methods are thin wrappers over `collective_rpc("<method_name>", args=...)`. The named method (`"execute_model"`, `"sample_tokens"`) is invoked on *every* worker in the group; the executor is a fan-out primitive.
- The return *shown* is the **abstract** base's `return output[0]`, but both concrete executors override these methods to request the reply from a single worker. `UniProcExecutor` uses `collective_rpc(..., single_value=True)` ([`uniproc_executor.py:108-131`](https://github.com/vllm-project/vllm/blob/6cf7b26bd4bff60bf378e1af14044280ac0d214c/vllm/v1/executor/uniproc_executor.py#L108-L131)); `MultiprocExecutor` uses `unique_reply_rank=self.output_rank` ([`multiproc_executor.py:310-332`](https://github.com/vllm-project/vllm/blob/6cf7b26bd4bff60bf378e1af14044280ac0d214c/vllm/v1/executor/multiproc_executor.py#L310-L332)), where `output_rank = _get_output_rank()` = `world_size − tp_size × prefill_context_parallel_size` — the first TP worker of the *last PP stage* ([`multiproc_executor.py:498-512`](https://github.com/vllm-project/vllm/blob/6cf7b26bd4bff60bf378e1af14044280ac0d214c/vllm/v1/executor/multiproc_executor.py#L498-L512)). That equals global rank 0 only for pure TP or a single GPU. Under tensor parallelism every rank runs the same forward pass over a sharded weight matrix and produces the same logits after the all-reduce, so any TP rank's `ModelRunnerOutput` is authoritative; under pipeline parallelism the authoritative output lives on the last stage, and the executor asks only that one rank for it rather than gathering all and discarding.
- `non_block=True` is why [Section 6](#6-the-engine-step-schedule-execute-process-outputs)'s overlap works: `collective_rpc` returns a `Future` instead of a materialized `ModelRunnerOutput`, so `EngineCore.step()` can run `get_grammar_bitmask` on the CPU while the GPU forward pass is in flight, then call `future.result()` as the sync point.

**Which concrete executor?** Chosen once, at construction, by `Executor.get_class` from `distributed_executor_backend`:

[`vllm/v1/executor/abstract.py:69-76`](https://github.com/vllm-project/vllm/blob/6cf7b26bd4bff60bf378e1af14044280ac0d214c/vllm/v1/executor/abstract.py#L69-L76)

```python
        elif distributed_executor_backend == "mp":
            from vllm.v1.executor.multiproc_executor import MultiprocExecutor

            executor_class = MultiprocExecutor
        elif distributed_executor_backend == "uni":
            from vllm.v1.executor.uniproc_executor import UniProcExecutor

            executor_class = UniProcExecutor
```

For a single GPU the backend resolves to `UniProcExecutor`, whose `collective_rpc` just calls the method on one in-process worker. For tensor/pipeline parallelism it resolves to `MultiprocExecutor` (the docs call it `MultiProcExecutor`), which relays the RPC to worker subprocesses; a Ray-based backend also exists and is deferred wholesale to article 11. The architecture docs summarize the same split — `MultiProcExecutor` for tensor parallelism, `UniProcExecutor` for single-GPU (<https://docs.vllm.ai/en/stable/design/arch_overview/>).


### From `SchedulerOutput` to a forward pass: the model runner

Each worker owns one model runner, and each model runner wraps the model's `torch.nn.Module` (<https://docs.vllm.ai/en/stable/design/arch_overview/>). When `collective_rpc("execute_model", ...)` lands on a worker, `GPUModelRunner.execute_model` first reconciles the worker's persistent batch with the scheduler's plan.

**Source anchor.** [`vllm/v1/worker/gpu/model_runner.py:1122-1133`](https://github.com/vllm-project/vllm/blob/6cf7b26bd4bff60bf378e1af14044280ac0d214c/vllm/v1/worker/gpu/model_runner.py#L1122-L1133) (excerpt; the full reconciliation is article 09's).

```python
            self.finish_requests(scheduler_output)
            ...
            self.add_requests(scheduler_output)
            self.update_requests(scheduler_output)
```


- The worker holds a *persistent* batch across steps and simply **applies the scheduler's diff**: it does not receive freshly-built tensors each iteration but drops finished requests, admits new ones, and refreshes positions/block tables for the survivors. This is V1's "persistent / differential batch," replacing V0's per-step tensor recreation (<https://vllm.ai/blog/2025-01-27-v1-alpha-release>). A guard elided above (`total_num_scheduled_tokens == 0`) short-circuits to `kv_connector.no_forward(...)` so a step scheduled purely for KV-connector bookkeeping runs no model. The full reconciliation set — `finish_requests`/`free_states`/`add_requests`/`update_requests`/`apply_staged_writes` — is article 09's.
- After this prologue, the runner builds the flattened input batch and runs the module. The tokens scheduled this step are packed into one flattened "super sequence"; position indices and attention metadata keep each sequence attending only to its own tokens, giving continuous batching without right-padding (<https://vllm.ai/blog/2025-09-05-anatomy-of-vllm>). *Input-ID/positions/slot-mapping/block-table construction and CUDA-graph capture/replay are article 09; the attention kernels that consume the block tables are article 08.*

`SchedulerOutput` is the source of truth for batch membership. The worker keeps a materialized local batch and updates it from diffs; sending full tensors every step would move CPU-side tensor construction back onto the execution path.

### The hinge: hidden states become logits

A forward pass produces hidden states for the scheduled tokens. The model runner selects the positions needed for sampling before projecting to the vocabulary.

**Source anchor.** [`vllm/v1/worker/gpu/model_runner.py:1054-1058`](https://github.com/vllm-project/vllm/blob/6cf7b26bd4bff60bf378e1af14044280ac0d214c/vllm/v1/worker/gpu/model_runner.py#L1054-L1058).

```python
        sample_hidden_states = hidden_states[input_batch.logits_indices]
        logits = self.model.compute_logits(sample_hidden_states)
        if grammar_output is not None:
            # Apply grammar bitmask to the logits in-place.
            assert self.structured_outputs_worker is not None
```


- `hidden_states` has one row per scheduled token in the flattened batch. `input_batch.logits_indices` picks the rows that need a next-token distribution — the final position of each sequence, plus extra positions when speculative draft tokens are being verified. This is the whole point of computing logits lazily: the vocab projection (`compute_logits`, a `hidden_dim × vocab_size` matmul) runs on a handful of rows, not on every prompt token in a long prefill.
- `compute_logits` applies the language-model head (and, under tensor parallelism, gathers the sharded vocabulary) to produce `[num_sampled_positions, vocab_size]` logits.
- If the scheduler attached a grammar bitmask for this step, it is applied to `logits` **in place**, right here, before sampling. This closes the loop opened in [Section 6](#6-the-engine-step-schedule-execute-process-outputs): `get_grammar_bitmask` was computed on the engine-core CPU thread concurrently with the forward pass, threaded through as `grammar_output`, and lands on the exact batch it was scheduled for. Structured-output constraints are enforced by masking logits, not by post-hoc filtering of tokens.

The runner projects the final positions needed for sampling, plus draft-verification positions when applicable, rather than every hidden state in the flattened batch. The grammar bitmask is passed to the sampling call associated with those logits.

### The execute/sample seam

Some execution paths return sampled tokens with the forward result. Others retain the execution state and let EngineCore call `sample_tokens(grammar_output)` after the CPU-produced grammar mask is ready. Section 9 follows that call; articles 10 and 12 cover ordinary and speculative sampling.

Source: [`vllm/v1/worker/gpu/model_runner.py:1358-1371`](https://github.com/vllm-project/vllm/blob/6cf7b26bd4bff60bf378e1af14044280ac0d214c/vllm/v1/worker/gpu/model_runner.py#L1358-L1371) and [`model_runner.py:1392-1394`](https://github.com/vllm-project/vllm/blob/6cf7b26bd4bff60bf378e1af14044280ac0d214c/vllm/v1/worker/gpu/model_runner.py#L1392-L1394)

```python
    def sample_tokens(
        self, grammar_output: GrammarOutput | None
    ) -> AsyncOutput | ModelRunnerOutput | None:
        if self.execute_model_state is None:
            # The prior execute_model call must have failed.
            return None

        input_batch = self.execute_model_state.input_batch
        attn_metadata = self.execute_model_state.attn_metadata
        slot_mappings_by_layer = self.execute_model_state.slot_mappings_by_layer
        hidden_states = self.execute_model_state.hidden_states
        aux_hidden_states = self.execute_model_state.aux_hidden_states
        finished_req_ids = self.execute_model_state.finished_req_ids
        self.execute_model_state = None
```

```python
        sampler_output, num_sampled, num_rejected = self.sample(
            hidden_states, input_batch, grammar_output
        )
```

`execute_model` stores the batch, hidden states, attention metadata, and finished-request set in `execute_model_state` when it defers sampling. `sample_tokens` retrieves that state and clears the field before calling `self.sample(...)`, making the stored state single-use for that execution step. The grammar output prepared by EngineCore enters the ordinary sampler at this point; when draft tokens are present, the same method later selects the rejection-sampling path described in article 12.

### What comes back

`sample_tokens` packages the selected IDs into a `ModelRunnerOutput`. A step may return more than one token for a request when speculative decoding or multi-token prediction is active, so downstream code treats new tokens as a list. `scheduler.update_from_output` records the result and retires finished requests.

The executor and distributed collectives continue in article 11; tensor preparation and CUDA graphs in article 09; PagedAttention in article 08; sampling policy in article 10; and rejection sampling in article 12. Here we only need the connecting path: the worker turns a scheduler diff into a flattened forward pass, projects the required positions to logits, applies the grammar mask when present, and returns sampled IDs for EngineCore to commit.

## 9. From Hidden States to a Sampled Token (Overview)

[Section 8](#8-model-execution-from-executor-to-logits-overview) ended with hidden states projected to vocabulary logits. V1 can cross the remaining logits-to-token gap in a separate, deferrable call. Article 10 covers penalties, temperature, top-k/top-p, min-p, and logprobs; this section follows the call boundary and the sampled ID.

### Sampling is a separate call, on purpose

In the engine step, `execute_model` runs with `non_block=True` and returns a `Future`. When that future resolves to `None`, the executor has deferred sampling, so EngineCore calls `sample_tokens` with the grammar bitmask prepared while the forward pass was in flight.

Source: [`vllm/v1/engine/core.py:497-499`](https://github.com/vllm-project/vllm/blob/6cf7b26bd4bff60bf378e1af14044280ac0d214c/vllm/v1/engine/core.py#L497-L499)

```python
            model_output = future.result()
            if model_output is None:
                model_output = self.model_executor.sample_tokens(grammar_output)
```


- **`future.result()` is the forward-pass sync point.** It returns a `ModelRunnerOutput` if the worker sampled inline, or `None` if the worker computed hidden states but stopped short of turning them into tokens.
- **`None` is the deferral sentinel.** It says "the forward pass is done; the logits exist; ask me to sample separately." The second call, `sample_tokens(grammar_output)`, feeds in the structured-output mask — the `grammar_output` computed one line earlier by `self.scheduler.get_grammar_bitmask(...)` ([Section 6](#6-the-engine-step-schedule-execute-process-outputs)), concurrently with the forward pass.
- **The mask meets the logits at sampling time.** Because the bitmask is built on the CPU while the GPU runs, and applied only in this `sample_tokens` call, it lands on *this* step's logits, for *this* step's requests.

**Invariant.** A structured-output bitmask is carried into the sampling operation for the scheduler step that produced it. EngineCore prepares the mask while the non-blocking model future is in flight; if execution already returned sampled output, the separate `sample_tokens` call is skipped.

### Across the executor boundary, symmetric to `execute_model`

`sample_tokens` crosses the same executor boundary as `execute_model` and uses the same `collective_rpc` shape. GPU topology remains the executor's concern.

Source: [`vllm/v1/executor/abstract.py:241-247`](https://github.com/vllm-project/vllm/blob/6cf7b26bd4bff60bf378e1af14044280ac0d214c/vllm/v1/executor/abstract.py#L241-L247)

```python
    def sample_tokens(
        self, grammar_output: GrammarOutput | None, non_block: bool = False
    ) -> ModelRunnerOutput | Future[ModelRunnerOutput]:
        output = self.collective_rpc(  # type: ignore[call-overload]
            "sample_tokens", args=(grammar_output,), non_block=non_block
        )
        return output[0]
```


- **`collective_rpc("sample_tokens", ...)` fans the call to the worker group** and the *abstract* base returns `output[0]`. Both concrete executors request a reply from one worker: `MultiprocExecutor` uses the first TP worker of the last PP stage ([`multiproc_executor.py:310-332`](https://github.com/vllm-project/vllm/blob/6cf7b26bd4bff60bf378e1af14044280ac0d214c/vllm/v1/executor/multiproc_executor.py#L310-L332), `498-512`), while `UniProcExecutor` uses `single_value=True` ([`uniproc_executor.py:108-131`](https://github.com/vllm-project/vllm/blob/6cf7b26bd4bff60bf378e1af14044280ac0d214c/vllm/v1/executor/uniproc_executor.py#L108-L131)). Forward and sample therefore use the same RPC structure.
- **Each worker lands in `GPUModelRunner.sample_tokens`** ([`model_runner.py:1358-1363`](https://github.com/vllm-project/vllm/blob/6cf7b26bd4bff60bf378e1af14044280ac0d214c/vllm/v1/worker/gpu/model_runner.py#L1358-L1363)), which retrieves the hidden states and `InputBatch` it stashed during `execute_model` (`self.execute_model_state`) and, on the last pipeline-parallel rank, calls `self.sample(hidden_states, input_batch, grammar_output)` ([`model_runner.py:1391-1394`](https://github.com/vllm-project/vllm/blob/6cf7b26bd4bff60bf378e1af14044280ac0d214c/vllm/v1/worker/gpu/model_runner.py#L1391-L1394)). Non-last PP ranks instead *receive* the sampled tokens broadcast from the last rank — sampling happens once, where the final hidden states live.

Sampling runs where the hidden states reside, behind the same executor interface as the forward pass. The executor handles TP/PP dispatch and reply selection; articles 09 and 11 cover those details. From EngineCore's perspective, logits-to-token is another `collective_rpc` operation.

### The hinge: hidden states to logits to a sampled token

Inside the model runner, one method turns the forward pass's hidden states into a `SamplerOutput`. This is the code boundary between model execution and sampling.

Source: [`vllm/v1/worker/gpu/model_runner.py:1048-1068`](https://github.com/vllm-project/vllm/blob/6cf7b26bd4bff60bf378e1af14044280ac0d214c/vllm/v1/worker/gpu/model_runner.py#L1048-L1068)

```python
    def sample(
        self,
        hidden_states: torch.Tensor,
        input_batch: InputBatch,
        grammar_output: GrammarOutput | None,
    ) -> tuple[SamplerOutput, torch.Tensor, torch.Tensor]:
        sample_hidden_states = hidden_states[input_batch.logits_indices]
        logits = self.model.compute_logits(sample_hidden_states)
        if grammar_output is not None:
            # Apply grammar bitmask to the logits in-place.
            assert self.structured_outputs_worker is not None
            self.structured_outputs_worker.apply_grammar_bitmask(
                logits,
                input_batch,
                grammar_output.structured_output_request_ids,
                grammar_output.grammar_bitmask,
            )

        if input_batch.num_draft_tokens == 0 or self.rejection_sampler is None:
            assert self.sampler is not None
            sampler_output = self.sampler(logits, input_batch)
```


- **`hidden_states[input_batch.logits_indices]` selects only the positions that need a token.** The forward pass ran over a flattened batch of *all* scheduled tokens ([Section 8](#8-model-execution-from-executor-to-logits-overview)), but only the last position of each sequence (and, under spec decode, a few extra) needs a logit. `logits_indices` gathers exactly those rows, so the vocab projection is done for a handful of positions, not the whole flattened batch.
- **`self.model.compute_logits(sample_hidden_states)`** is the hidden-states-to-vocab projection — the language-model head. This single line is the boundary the whole section is named after.
- **The grammar bitmask is applied in place, right here.** This closes the loop from [Section 6](#6-the-engine-step-schedule-execute-process-outputs): the mask is scheduler-owned, carried across `sample_tokens`, and stamped onto the logits *before* the sampler ever sees them — so a structured-output request cannot sample a token its grammar forbids.
- **`self.sampler(logits, input_batch)`** is the handoff into the sampler proper (the common, no-spec-decode branch). The `else` branch ([`model_runner.py:1069-1078`](https://github.com/vllm-project/vllm/blob/6cf7b26bd4bff60bf378e1af14044280ac0d214c/vllm/v1/worker/gpu/model_runner.py#L1069-L1078)) routes to `self.rejection_sampler` when draft tokens are present — speculative decoding, deferred to article 12.

The `[logits_indices]` gather projects the positions that need logits instead of constructing a full `[num_all_tokens, vocab_size]` tensor for the flattened batch. In the ordinary decode case, that is the final position for each request; speculative verification may select additional positions.

### The sampler entry point

`self.sampler(...)` invokes `Sampler.__call__`, which prepares per-request views and delegates the token draw to `Sampler.sample`. The operations before that draw form the logits-processing pipeline.

Source: [`vllm/v1/worker/gpu/sample/sampler.py:198-216`](https://github.com/vllm-project/vllm/blob/6cf7b26bd4bff60bf378e1af14044280ac0d214c/vllm/v1/worker/gpu/sample/sampler.py#L198-L216) (excerpt; the full pipeline is article 10's):

```python
    def sample(
        self,
        logits: torch.Tensor,
        ...
    ) -> tuple[torch.Tensor, torch.Tensor]:
        processed_logits = self.apply_sampling_params(
            logits,
            ...
            skip_top_k_top_p=True,
        )
```

<p class='figure-caption'>From hidden states to a token id: `hidden_states[input_batch.logits_indices]` selects the positions that need logits, `compute_logits` projects them to vocab, a scheduler-owned grammar bitmask is applied in place, then `apply_sampling_params` runs its operators in order (bias / `min_tokens` → penalties → bad-words → temperature → min-p, with top-k/top-p deferred) on a private fp32 copy before the draw — so sampling cost scales with the number of selected logits rows: roughly one per request in ordinary decode, and more when speculative verification scores several positions per request.</p>


- **`Sampler.__call__` ([`sampler.py:72-102`](https://github.com/vllm-project/vllm/blob/6cf7b26bd4bff60bf378e1af14044280ac0d214c/vllm/v1/worker/gpu/sample/sampler.py#L72-L102)) is the true entry from the model runner on the ordinary no-spec-decode path.** It derives `pos` and `input_ids` from `input_batch.logits_indices`, decides whether logprobs are needed, calls `self.sample(...)`, and packs the result into a `SamplerOutput` whose `sampled_token_ids` is reshaped to `[num_requests, 1]` ([`sampler.py:134-144`](https://github.com/vllm-project/vllm/blob/6cf7b26bd4bff60bf378e1af14044280ac0d214c/vllm/v1/worker/gpu/sample/sampler.py#L134-L144)) — one sampled token per request on this branch. Requests with draft tokens take the separate rejection-sampler branch described above.
- **`apply_sampling_params` is the logits-processing pipeline.** In order ([`sampler.py:146-196`](https://github.com/vllm-project/vllm/blob/6cf7b26bd4bff60bf378e1af14044280ac0d214c/vllm/v1/worker/gpu/sample/sampler.py#L146-L196)): copy logits to a fresh fp32 tensor, then apply logit bias / `allowed_token_ids` / `min_tokens`, penalties (presence, frequency, repetition), bad-words masking, temperature, and min-p — with top-k/top-p deferred here (`skip_top_k_top_p=True`) so `sample` can route them through the FlashInfer fused path or the fallback. Each of those operators is a section of article 10; article 01 states only their existence and order.
- **The draw itself** ([`sampler.py:217-244`](https://github.com/vllm-project/vllm/blob/6cf7b26bd4bff60bf378e1af14044280ac0d214c/vllm/v1/worker/gpu/sample/sampler.py#L217-L244)) picks FlashInfer's fused top-k/top-p sampler when eligible, else applies top-k/top-p and calls `gumbel_sample`, returning `(sampled, processed_logits)`.


### Closing the loop: the token id becomes `new_token_ids`

The sampled ID does not return directly to the caller as an integer. It travels in `EngineCoreOutput` and is folded into request progress by `update_from_output`.

Source: [`vllm/v1/engine/__init__.py:181-205`](https://github.com/vllm-project/vllm/blob/6cf7b26bd4bff60bf378e1af14044280ac0d214c/vllm/v1/engine/__init__.py#L181-L205) (two key fields; [Section 10](#10-output-processing-enginecoreoutput-to-requestoutput) quotes the full struct):

```python
    request_id: str
    new_token_ids: list[int]
    ...
    @property
    def finished(self) -> bool:
        return self.finish_reason is not None
```


- **`new_token_ids: list[int]`, not `int`.** The path from sampler to caller is multi-token-ready from the first field. In the common case the list holds one id; under speculative decoding or other multi-token schemes it holds several accepted ids for one step (article 12). Nothing downstream — detokenizer, stop-string check, `RequestOutput` — has to special-case spec decode at this boundary because the type already accounts for it.
- **`finished` is a derived property, not a stored flag.** It is `True` iff `finish_reason` is set. The engine core sets `finish_reason` on length/EOS finishes; stop-*string* finishes are decided later, in the output processor, over detokenized text ([Section 10](#10-output-processing-enginecoreoutput-to-requestoutput)) — which is why `EngineCoreOutput.finished` and the caller-visible finish can legitimately differ for one step.
- **`num_nans_in_logits`** is the corruption tripwire: the sampler computes it *before* penalties and temperature ([`sampler.py:84-86`](https://github.com/vllm-project/vllm/blob/6cf7b26bd4bff60bf378e1af14044280ac0d214c/vllm/v1/worker/gpu/sample/sampler.py#L84-L86)) so a non-zero value points at the model's own output, not at sampling.

The sampled token leaves the worker as data in an engine output rather than as a direct return to the public caller. `update_from_output` reconciles it with request state before `EngineCoreOutputs` travels through the client. The first non-empty `new_token_ids` for a request marks the event that output processing can turn into the first visible token.

The `rejection_sampler` branch and the multi-token `new_token_ids` are **article 12** (Speculative Decoding). The `logits_indices` gather and the flattened-batch layout it indexes into belong to **article 08/09**. The next section follows `new_token_ids` off the GPU and into text.

## 10. Output Processing: EngineCoreOutput to RequestOutput

Every prior section pushed a request *forward*: the façade normalized it, the input processor wire-shaped it, the client abstraction ferried it across a process boundary, the engine step scheduled and sampled it. This is the one section that runs the other way. Sampled token IDs leave `EngineCore.step()` as `EngineCoreOutputs` — a per-client bundle of per-request `EngineCoreOutput`s — and have to become user-facing `RequestOutput`s carrying decoded text, finish reasons, and logprobs. That translation is the `OutputProcessor`, and it is deliberately *not* part of the hot engine loop. [Section 6](#6-the-engine-step-schedule-execute-process-outputs) already showed the seam from the caller's side: `LLMEngine.step()` calls `get_output()`, then `output_processor.process_outputs(...)`, then feeds the returned `reqs_to_abort` back into the engine. This section opens that middle call and shows the contracts it defends.

### The input struct: what the engine actually ships back

The boundary carries token IDs and optional metadata, not text.

Source: [`vllm/v1/engine/__init__.py:175-205`](https://github.com/vllm-project/vllm/blob/6cf7b26bd4bff60bf378e1af14044280ac0d214c/vllm/v1/engine/__init__.py#L175-L205)

```python
class EngineCoreOutput(
    msgspec.Struct,
    array_like=True,  # type: ignore[call-arg]
    omit_defaults=True,  # type: ignore[call-arg]
    gc=False,
):  # type: ignore[call-arg]
    request_id: str
    new_token_ids: list[int]
    ...
    finish_reason: FinishReason | None = None
    stop_reason: int | str | None = None
    ...
    @property
    def finished(self) -> bool:
        return self.finish_reason is not None
```

`new_token_ids` is a **list**, not a scalar — one step can commit more than one token per request (speculative decoding, multi-token prediction), so the return path must be written to fold *N* tokens at once rather than assuming one. Everything after it is optional (`omit_defaults=True`), and `finished` is a *derived* property: a request is finished exactly when the engine attached a `finish_reason` (length, EOS, stop-token). Notably there is **no text field** — detokenization was moved out of the engine process in V1 so it overlaps GPU work ([V1 blog](https://vllm.ai/blog/2025-01-27-v1-alpha-release)). The engine ships IDs; text is the output processor's job.

`req_state` is the long-lived per-request accumulator (`RequestState`) holding this request's detokenizer, logprobs processor, prompt data, and streaming bookkeeping. Its absence is **not an error** — a request aborted a moment ago (from a client disconnect, a cancelled `generate()`, a prior stop string) has already been popped from `request_states`, and the engine may still be mid-flight producing one last output for it. Silently `continue`-ing is what makes abort and output processing race-safe without a lock: the two sides converge because a missing entry is defined to mean "drop," not "crash."

If `request_states` no longer contains a request, the output processor discards a late output for it. That behavior allows an abort to race with an already in-flight step without resurrecting frontend state.

The detokenizer's `update` is where token IDs become characters. Its subtlety is that the *token* stream and the *text* stream are deliberately allowed to disagree.

When the engine stopped on a stop *token* and the user did not ask to keep it, the last ID is pulled out of the decode set (`skipped_stop_token_id`) so it contributes no characters, then re-appended to `token_ids` — the token list stays complete while the text omits the terminator. The elided incremental loop decodes each new id and advances `stop_check_offset` (a `min_tokens`-gated high-water mark) so the stop-string search covers only *newly added* characters — matching is O(new chars), not O(whole output), each step; the full loop is article 10's. The `num_output_tokens() > self.min_tokens` guard gates the whole stop block, so **no stop string can fire before `min_tokens` is reached**. On a match, `check_stop_strings` may return a truncation length so `output_text` is cut to drop the stop string (unless `truncate_to == -1`, meaning it sits at the very end and no cut is needed).

Back in the loop: if `update` returns a stop string, the processor **overwrites `finish_reason` to `STOP` and sets `stop_reason` to the matched string**. This is the divergence the input struct set up: `engine_core_output.finished` may still be `False` (the engine kept generating) while the processor has locally decided the request is done. [Section 6](#6-the-engine-step-schedule-execute-process-outputs) showed the other half of the closure — the loop appends such `req_id`s to `reqs_to_abort`, and `LLMEngine.step()` / the async `output_handler` feed them back so the *next* `EngineCore.step()` stops that request at its abort window.

**Invariant.** `token_ids` retains the sampled sequence even when `output_text` omits a stop token or stop string. Text-side stop handling respects `min_tokens`; when it finishes a request that EngineCore still considers live, it returns that request through `reqs_to_abort`.

`output_kind` comes from `SamplingParams.output_kind` and selects the surfacing contract: **`FINAL_ONLY`** (offline default — [Section 2](#2-llmgenerate-the-offline-convenience-layer) showed `_add_request` forcing it) emits exactly one output, on completion, so every non-final step returns `None`; **`DELTA`** (OpenAI streaming) emits only the newly generated slice; **`CUMULATIVE`** re-sends the growing whole. `stream_interval > 1` further coalesces: emit only on finish, on the first token, or every `stream_interval` tokens — and in DELTA mode it re-slices `new_token_ids` to `output_token_ids[sent_tokens_offset:]` and advances `sent_tokens_offset`, so tokens skipped across quiet steps are delivered together rather than dropped. ## 11. Streaming, the Async Generator, and Why the First Token Is Special

The offline path ends in a blocking `while has_unfinished_requests(): step()` loop that accumulates finished outputs and sorts them ([`offline_utils.py:594-599`](https://github.com/vllm-project/vllm/blob/6cf7b26bd4bff60bf378e1af14044280ac0d214c/vllm/entrypoints/offline_utils.py#L594-L599)). Online serving instead needs incremental delivery without coupling engine progress to one HTTP consumer. V1 keeps the engine step and changes the return path: each per-step batch is fanned into per-request mailboxes drained by async generators.

The online return path is **two producer/consumer boundaries stacked and decoupled by asyncio**. Boundary A: a background task owns the ZMQ output socket, decodes each frame into an `EngineCoreOutputs` batch (one per engine step, carrying the requests that produced output that step — not necessarily every in-flight request), and pushes it onto `AsyncMPClient.outputs_queue`, an `asyncio.Queue` ([`core_client.py:973`](https://github.com/vllm-project/vllm/blob/6cf7b26bd4bff60bf378e1af14044280ac0d214c/vllm/v1/engine/core_client.py#L973), `1005-1062`). Boundary B: a single `output_handler` task per `AsyncLLM` awaits that queue, runs `OutputProcessor.process_outputs`, and dispatches each per-request result into its own mailbox. The socket topology behind Boundary A is article 03's territory; here we care about what crosses Boundary B and reaches the caller.

### `generate()` is a pure consumer

The public online API, `AsyncLLM.generate`, is an async generator. Crucially it does **not** drive the engine — it only consumes its own mailbox.

<p class='figure-caption'>The online return path is two asyncio-decoupled boundaries: Boundary A (a background task decodes ZMQ frames into `EngineCoreOutputs` and pushes them onto `outputs_queue`) and Boundary B (one `output_handler` runs `process_outputs` and dispatches each result into a per-request single-slot, coalescing `RequestOutputCollector`), which `generate()` drains with `q.get_nowait() or await q.get()` — at most one pending output per request; coalescing preserves the logical output content even as it merges intermediate event boundaries.</p>


1. `q` is the per-request `RequestOutputCollector` returned by `add_request` ([`async_llm.py:559`](https://github.com/vllm-project/vllm/blob/6cf7b26bd4bff60bf378e1af14044280ac0d214c/vllm/v1/engine/async_llm.py#L559)), which also registered the request with the `OutputProcessor` and submitted it to EngineCore in the same await-chain ([Section 5](#5-add_request-a-request-enters-enginecore)). By the time this loop runs, the mailbox is already wired into the fan-out.
2. `out = q.get_nowait() or await q.get()` (L579) tries the non-blocking grab first; only an empty mailbox forces an `await`. The comment (L577-578) states the intent: skipping the event-loop task switch matters under load, because at high concurrency a token is usually already waiting.
3. The terminal condition rides *on the data*: `finished = out.finished` (L584). The producer stamps `finished=True` on the last `RequestOutput`; the consumer stops after yielding it. `generate()` needs zero visibility into scheduler or EngineCore lifecycle to know when to stop.
4. `STREAM_FINISHED` (a sentinel `RequestOutput` with `finished=True` and empty `outputs`, [`outputs.py:191-199`](https://github.com/vllm-project/vllm/blob/6cf7b26bd4bff60bf378e1af14044280ac0d214c/vllm/outputs.py#L191-L199)) is filtered out (L585) — it unblocks the loop on streaming-input completion without emitting a spurious empty chunk.


### The per-request mailbox is a single-slot, coalescing buffer

The hand-off from the shared `output_handler` (producer) to the per-request `generate()` (consumer) is deliberately *not* an unbounded queue. It is a single-slot mailbox plus an `asyncio.Event`, and if the producer outruns the consumer, successive outputs are **merged in place**.

[`vllm/v1/engine/output_processor.py:62-86`](https://github.com/vllm-project/vllm/blob/6cf7b26bd4bff60bf378e1af14044280ac0d214c/vllm/v1/engine/output_processor.py#L62-L86):

```python
    def put(self, output: RequestOutput | PoolingRequestOutput | Exception) -> None:
        """Non-blocking put operation."""
        if self.output is None or isinstance(output, Exception):
            self.output = output
            self.ready.set()
        elif isinstance(self.output, RequestOutput) and isinstance(
            output, RequestOutput
        ):
            # This ensures that request outputs with different request indexes
            # (if n > 1) do not override each other.
            self.output.add(output, aggregate=self.aggregate)
        elif isinstance(self.output, PoolingRequestOutput) and isinstance(
            output, PoolingRequestOutput
        ):
            self.output = output

    async def get(self) -> RequestOutput | PoolingRequestOutput:
        """Get operation blocks on put event."""
        while (output := self.output) is None:
            await self.ready.wait()
        self.output = None
        self.ready.clear()
        if isinstance(output, Exception):
            raise output
        return output
```


1. `put()` runs synchronously from the producer. If the slot is empty (or the item is an `Exception`), install it and `self.ready.set()` — an exception always wins the slot immediately and is never coalesced away, so a dead engine surfaces to the waiter at once.
2. If the slot already holds a `RequestOutput`, the new one is folded in via `self.output.add(output, aggregate=self.aggregate)` (L72) rather than overwriting. `aggregate` is `True` only in `DELTA` mode (`self.aggregate = output_kind == RequestOutputKind.DELTA`, L55). The merge ([`outputs.py:145-173`](https://github.com/vllm-project/vllm/blob/6cf7b26bd4bff60bf378e1af14044280ac0d214c/vllm/outputs.py#L145-L173)) concatenates text/token-ids/logprobs for DELTA streaming, replaces the whole snapshot for cumulative streaming, and OR-folds `finished` so a coalesced batch that swallowed the terminal chunk still stops the generate loop.
3. `get()` awaits `self.ready` until the slot is non-`None`, then clears both slot and event.

The mailbox holds at most one pending `RequestOutput` object per request. In DELTA mode, merging concatenates token and text deltas; in cumulative mode, the newer snapshot replaces the older one. This bounds the mailbox object count without discarding the logical output represented by the merged value.

### One loop, one flag: where sync and async fork

The synchronous and asynchronous paths reuse the same hot CPU loop over the batch — `process_outputs`, "the only function that should loop over EngineCoreOutputs" ([`output_processor.py:594-601`](https://github.com/vllm-project/vllm/blob/6cf7b26bd4bff60bf378e1af14044280ac0d214c/vllm/v1/engine/output_processor.py#L594-L601)). [Section 10](#10-output-processing-enginecoreoutput-to-requestoutput) already opened the one branch inside it that forks sync from async: `if req_state.queue is not None:` pushes the materialized `RequestOutput` into the per-request mailbox (async) instead of appending it to a returned list (sync) ([`output_processor.py:650-666`](https://github.com/vllm-project/vllm/blob/6cf7b26bd4bff60bf378e1af14044280ac0d214c/vllm/v1/engine/output_processor.py#L650-L666)). The async engine registers that `queue` at admission ([Section 5](#5-add_request-a-request-enters-enginecore)); the sync engine passes `queue=None`.

What is genuinely [Section 11](#11-streaming-the-async-generator-and-why-the-first-token-is-special)'s own is that fork's consequence for streaming. Because a queue is present, async dispatch is provably *push-only* — which is why the `output_handler` can `assert not processed_outputs.request_outputs` ([`async_llm.py:679`](https://github.com/vllm-project/vllm/blob/6cf7b26bd4bff60bf378e1af14044280ac0d214c/vllm/v1/engine/async_llm.py#L679)): with a queue wired in, the return list is never touched. So streaming adds *no second loop* over the batch — it reuses the one offline already runs and swaps only the last statement. The detokenization, stop-string checks, and logprob assembly that `make_request_output` depends on are article 10's subject; article 04 details the finished-request bookkeeping and the `reqs_to_abort` path.

### Why the first token is special

Prefill and decode are two different forward passes. From the "Inside vLLM" walkthrough ([Inside vLLM](https://vllm.ai/blog/2025-09-05-anatomy-of-vllm)): **prefill** is "a forward pass over all prompt tokens," typically compute-bound; **decode** is "a forward pass over just the most recent token," memory-bandwidth-bound. V1 does not run them as separate phases — it treats all tokens uniformly and a scheduling decision is conceptually `{request_id: num_tokens}` ([V1 blog](https://vllm.ai/blog/2025-01-27-v1-alpha-release)), so a long prompt can be split across several steps (chunked prefill, article 05). The consequence for streaming: a request can occupy the GPU for one or more steps producing *no* sampled token, because only the final prefill position yields logits to sample. The first `EngineCoreOutput` that carries a real `new_token_ids` is therefore a distinct event, and the output processor marks it.

[`vllm/v1/engine/output_processor.py:628-633`](https://github.com/vllm-project/vllm/blob/6cf7b26bd4bff60bf378e1af14044280ac0d214c/vllm/v1/engine/output_processor.py#L628-L633):

```python

                if engine_core_output.prefill_stats is not None:
                    req_state.num_cached_tokens = (
                        engine_core_output.prefill_stats.num_cached_tokens

                req_state.is_prefilling = False
```

`is_prefilling` starts `True` at `RequestState` construction ([`output_processor.py:172`](https://github.com/vllm-project/vllm/blob/6cf7b26bd4bff60bf378e1af14044280ac0d214c/vllm/v1/engine/output_processor.py#L172)); a streaming-input update can set it to `True` again ([`output_processor.py:191-208`](https://github.com/vllm-project/vllm/blob/6cf7b26bd4bff60bf378e1af14044280ac0d214c/vllm/v1/engine/output_processor.py#L191-L208)). The first output captures `num_cached_tokens` and flips the flag to `False`. Without a later streaming-input prefill, subsequent outputs are decode updates. TTFT includes scheduler wait, any chunked-prefill steps, sampling, incremental detokenization, and return-path fan-out; inter-token latency measures the later decode iterations.

The full first-token chain across process boundaries is: `EngineCore.step()` commits the sampled token in `update_from_output` and returns a per-client `dict[int, EngineCoreOutputs]` ([`core.py:504-508`](https://github.com/vllm-project/vllm/blob/6cf7b26bd4bff60bf378e1af14044280ac0d214c/vllm/v1/engine/core.py#L504-L508)); Boundary A enqueues that batch; the `output_handler` runs `process_outputs`, which flips `is_prefilling` (above) and pushes the first `RequestOutput` into the mailbox; `generate()` yields it ([`async_llm.py:585-586`](https://github.com/vllm-project/vllm/blob/6cf7b26bd4bff60bf378e1af14044280ac0d214c/vllm/v1/engine/async_llm.py#L585-L586)). No single call stack spans that chain — it is a relay across the async client, the shared handler, the per-request mailbox, and the consumer task.

### Cancellation closes the loop

Because `generate()` is consumed by an HTTP client that can disconnect, cancellation must tear the request down on *both* sides or the engine keeps decoding a dead request. [`async_llm.py:591-596`](https://github.com/vllm-project/vllm/blob/6cf7b26bd4bff60bf378e1af14044280ac0d214c/vllm/v1/engine/async_llm.py#L591-L596) catches `CancelledError`/`GeneratorExit`, calls `self.abort(q.request_id, internal=True)` (which aborts in both the OutputProcessor and EngineCore), and re-raises; the `finally` at [`async_llm.py:633-635`](https://github.com/vllm-project/vllm/blob/6cf7b26bd4bff60bf378e1af14044280ac0d214c/vllm/v1/engine/async_llm.py#L633-L635) calls `q.close()` to cancel any streaming-input feeder. The blocking offline API has no per-client disconnect path analogous to this: `generate()` is a blocking call, not an iterator the caller can abandon, so cancellation there follows a different control path.

The streaming path reuses the same engine work with an async return channel. Offline `LLMEngine.step` processes outputs with `queue=None` and returns a list ([`llm_engine.py:296-334`](https://github.com/vllm-project/vllm/blob/6cf7b26bd4bff60bf378e1af14044280ac0d214c/vllm/v1/engine/llm_engine.py#L296-L334)); the async path pushes into request mailboxes. The scheduler, workers, and sampler are independent of that delivery choice. Article 02 covers the OpenAI server, article 03 the ZMQ client, and article 04 output handling and request teardown.

## 12. The Full End-to-End Trace

We have now crossed every boundary on the request path once. This closing section does two things: it reassembles those boundaries into a single logical thread — one uninterrupted trace from `LLM.generate()` to the `RequestOutput` in the caller's hands — and it distills the handful of principles worth carrying into the deep articles. Nothing here is new machinery; every anchor below was established in an earlier section. The value of reading them as one trace is that the *seams* become visible: you can see precisely where a request changes owner, changes representation, and changes process, and you can name the contract that each seam buys.

### The trace, hop by hop

Follow one prompt. The line anchors point at the exact code each hop executes.

1. **`LLM.generate()` validates and delegates** — [`vllm/entrypoints/llm.py:465-485`](https://github.com/vllm-project/vllm/blob/6cf7b26bd4bff60bf378e1af14044280ac0d214c/vllm/entrypoints/llm.py#L465-L485). The runner-type gate rejects non-generative models, `sampling_params` is defaulted so it is never `None` downstream, and control passes to `_run_completion`. No scheduling, no allocation. *(Article 02.)*

2. **The offline mixin stamps the request and admits it** — [`vllm/entrypoints/offline_utils.py:559-563`](https://github.com/vllm-project/vllm/blob/6cf7b26bd4bff60bf378e1af14044280ac0d214c/vllm/entrypoints/offline_utils.py#L559-L563). `output_kind` is forced to `FINAL_ONLY` and a monotonic decimal `request_id` is minted from a `Counter()`. Then `LLMEngine.add_request` ([`vllm/v1/engine/llm_engine.py:218-294`](https://github.com/vllm-project/vllm/blob/6cf7b26bd4bff60bf378e1af14044280ac0d214c/vllm/v1/engine/llm_engine.py#L218-L294)) registers the work on *both* sides — the output processor (for detokenization and aggregation) and the engine core (for scheduling) — and, when `n > 1`, fans one logical request into `n` children under a `ParentRequest`. *(Article 02 for the façade, 04 for the two-sided registration.)*

3. **The input processor turns a prompt into a wire struct** — [`vllm/v1/engine/input_processor.py:242-255`](https://github.com/vllm-project/vllm/blob/6cf7b26bd4bff60bf378e1af14044280ac0d214c/vllm/v1/engine/input_processor.py#L242-L255). `process_inputs(request_id, prompt, params, ...) -> EngineCoreRequest` runs the validation gauntlet, clones and finalizes the sampling params (so `max_tokens` is concrete), and flattens any multimodal inputs; it normally receives an upstream-rendered `EngineInput`, with raw-prompt tokenization surviving only on a deprecated compatibility path. Its output is an `EngineCoreRequest` — a compact `msgspec` struct designed to cross a socket. *(Article 03.)*

4. **The client abstraction moves it across the process boundary** — [`vllm/v1/engine/llm_engine.py:104-111`](https://github.com/vllm-project/vllm/blob/6cf7b26bd4bff60bf378e1af14044280ac0d214c/vllm/v1/engine/llm_engine.py#L104-L111). `LLMEngine` holds one polymorphic `EngineCoreClient`. In the multiprocess default the request is serialized behind a one-byte type tag ([`vllm/v1/engine/__init__.py:251-264`](https://github.com/vllm-project/vllm/blob/6cf7b26bd4bff60bf378e1af14044280ac0d214c/vllm/v1/engine/__init__.py#L251-L264): `ADD = b"\x00"`) and ROUTER-sent over ZMQ to a busy-looping `EngineCore` process; with the in-process client it is a direct method call. Same `add_request`/`get_output` surface either way. *(Article 03 for the ZMQ topology, 11 for DP routing.)*

5. **`EngineCore.step()` runs the transaction** — [`vllm/v1/engine/core.py:479-508`](https://github.com/vllm-project/vllm/blob/6cf7b26bd4bff60bf378e1af14044280ac0d214c/vllm/v1/engine/core.py#L479-L508). Schedule, launch the forward pass non-blocking, build the grammar bitmask on the CPU while the GPU runs, synchronize, drain aborts, then commit. This is the repeated heartbeat the whole system is built around, and step 6–9 all happen *inside* it. A tail hook, `post_step` ([`core.py:510-517`](https://github.com/vllm-project/vllm/blob/6cf7b26bd4bff60bf378e1af14044280ac0d214c/vllm/v1/engine/core.py#L510-L517)), feeds speculative draft tokens back to the scheduler for the next step. *(Article 04.)*

6. **The scheduler plans tokens, not tensors** — the single call `scheduler.schedule(...)` at [`core.py:490`](https://github.com/vllm-project/vllm/blob/6cf7b26bd4bff60bf378e1af14044280ac0d214c/vllm/v1/engine/core.py#L490) answers "which requests run, and how many tokens does each advance," expressed as the uniform counter model `{request_id: num_tokens}`. It returns a `SchedulerOutput` plan and touches no GPU. *(Article 05; KV allocation 06, prefix reuse 07, draft tokens 12.)*

7. **The executor fans the plan out to the GPU** — [`vllm/v1/executor/abstract.py:210-227`](https://github.com/vllm-project/vllm/blob/6cf7b26bd4bff60bf378e1af14044280ac0d214c/vllm/v1/executor/abstract.py#L210-L227). `execute_model` is a `collective_rpc` that names an *operation*, not a device, so one or N workers run the same call and one authoritative worker's answer is taken (the last PP stage's first TP rank; [Section 8](#8-model-execution-from-executor-to-logits-overview)). On each worker the model runner reconciles its persistent batch, runs the flattened forward pass, and at the hinge ([`vllm/v1/worker/gpu/model_runner.py:1054-1055`](https://github.com/vllm-project/vllm/blob/6cf7b26bd4bff60bf378e1af14044280ac0d214c/vllm/v1/worker/gpu/model_runner.py#L1054-L1055)) selects final-position hidden states and projects them to logits. *(Articles 08–09 for kernels/CUDA graphs, 11 for parallelism.)*

8. **The sampler draws token IDs** — either inline or, when the worker deferred it, via the second `collective_rpc`, `sample_tokens(grammar_output)` at [`core.py:497-499`](https://github.com/vllm-project/vllm/blob/6cf7b26bd4bff60bf378e1af14044280ac0d214c/vllm/v1/engine/core.py#L497-L499), reaching `Sampler.sample` ([`vllm/v1/worker/gpu/sample/sampler.py:198-210`](https://github.com/vllm-project/vllm/blob/6cf7b26bd4bff60bf378e1af14044280ac0d214c/vllm/v1/worker/gpu/sample/sampler.py#L198-L210)). The grammar bitmask is applied to *these* logits in place before drawing. *(Article 10; rejection sampling for spec decode is 12.)*

9. **The commit folds tokens into progress** — `scheduler.update_from_output(...)` at [`core.py:504-506`](https://github.com/vllm-project/vllm/blob/6cf7b26bd4bff60bf378e1af14044280ac0d214c/vllm/v1/engine/core.py#L504-L506) appends the sampled IDs, sets finish reasons, and starts freeing the KV blocks of finished requests — immediately when safe, or deferred behind an in-flight GPU step. Its result is a `dict[client_index -> EngineCoreOutputs]` — token IDs and finish flags, *not* user-facing text. *(Article 04.)*

10. **The output processor turns IDs back into text** — [`vllm/v1/engine/output_processor.py:576-693`](https://github.com/vllm-project/vllm/blob/6cf7b26bd4bff60bf378e1af14044280ac0d214c/vllm/v1/engine/output_processor.py#L576-L693), the one function allowed to loop over the whole batch. It detokenizes incrementally, promotes a detected stop *string* to a finish (and, if the core did not also stop, appends the id to `reqs_to_abort`), and materializes a `RequestOutput`. *(Articles 04 and 10.)*

11. **The result returns two ways.** Offline: the drain loop collects finished outputs and restores input order with a terminal sort ([`offline_utils.py:594-626`](https://github.com/vllm-project/vllm/blob/6cf7b26bd4bff60bf378e1af14044280ac0d214c/vllm/entrypoints/offline_utils.py#L594-L626)). Online: `process_outputs` pushes each `RequestOutput` into a per-request mailbox that the `generate()` async generator yields ([`vllm/v1/engine/async_llm.py:576-586`](https://github.com/vllm-project/vllm/blob/6cf7b26bd4bff60bf378e1af14044280ac0d214c/vllm/v1/engine/async_llm.py#L576-L586)). Both are the *same* `process_outputs` output, dispatched by a single `if req_state.queue is not None:` fork. *(Article 02 offline, 03/11 online streaming.)*

At the final boundary, the result recovers the caller's external request ID.

[`vllm/v1/engine/output_processor.py:363-364`](https://github.com/vllm-project/vllm/blob/6cf7b26bd4bff60bf378e1af14044280ac0d214c/vllm/v1/engine/output_processor.py#L363-L364):

```python
        return RequestOutput(
            request_id=external_req_id,  # request_id is what was provided externally
```

Internal request IDs may gain random suffixes, and parallel sampling may create child IDs. `RequestOutput` switches back to `external_req_id` before the result leaves output processing.

### Takeaways

- `LLM.generate()` is a synchronous API, not necessarily an in-process engine. With V1 multiprocessing enabled—the default at this baseline—it uses `SyncMPClient` and a background EngineCore process.
- The request changes representation as ownership changes: public input → `EngineCoreRequest` → scheduler `Request` → batched tensors → sampled IDs → `RequestOutput`.
- `EngineCore.step()` is the recurring center of the system. It schedules work, launches execution, handles the sampling seam, processes aborts, and commits output.
- The scheduler grants token work and owns request/cache lifecycle; the model runner owns tensor preparation and execution; the output processor owns detokenization and API-facing results.
- Offline and async callers share the same output-processing logic. They differ at delivery: a blocking drain with final outputs versus per-request async mailboxes.
