## 1. What Is FlatBuffers?

FlatBuffers is an open-source serialization library that implements a serialization format similar to Protocol Buffers, Thrift, Apache Avro, SBE, and Cap'n Proto. It was primarily written by Wouter van Oortmerssen and open-sourced by Google. Oortmerssen originally developed FlatBuffers for Android games and performance-sensitive applications. Today, it has ports for C++, C#, C, Go, Java, PHP, Python, and JavaScript.

A FlatBuffer is a binary buffer that uses offsets to organize nested objects (`struct`, `table`, `vectors`, and so on), allowing data to be accessed in place, much like any pointer-based data structure. Unlike most in-memory data structures, however, a FlatBuffer uses strict alignment rules and byte ordering to ensure that the buffer is cross-platform. In addition, for `table` objects, FlatBuffers provides forward/backward compatibility and optional fields to support schema evolution in most formats.

The primary goal of FlatBuffers is to avoid deserialization. It achieves this by defining a binary data protocol—a well-defined way to convert data into binary form. The binary structure created by this protocol can be sent over the wire and read without further processing. By comparison, when transmitting JSON, we need to convert data into a string, send it over the wire, parse the string, and convert it into local objects. FlatBuffers does not require these operations. You load data into binary form, send the same binary data, and read directly from the binary data.

Although FlatBuffers has its own interface definition language for defining the data to be serialized, it also supports the `.proto` format used by Protocol Buffers.

You define object types in a schema, and then compile them into mainstream languages such as C++ or Java to enable zero-overhead reads and writes. FlatBuffers also supports dynamically parsing JSON data into a buffer.

Beyond parsing efficiency, the binary format brings another advantage: binary representations of data are usually more efficient. For example, we can store a 10-digit integer using a 4-byte UInt instead of 10 characters.

## 2. Why Was FlatBuffers Invented?

<p align='center'>
<img src='../images/flatbuffers.png'>
</p>

JSON is a language-independent data format, but parsing it and converting it into objects such as Java objects consumes time and memory. Parsing a 20 KB JSON stream on the client takes roughly 35 ms, while a single UI refresh budget is 16.6 ms. In highly real-time games, any stutter or delay is unacceptable, so a new data format was needed. On the server side, parsing JSON can sometimes create a large number of small objects. For servers that need to process JSON data for millions of players per second, the pressure increases. If each JSON parse produces many small objects, the massive number of small objects generated by a massive number of players may cause GC-related issues during memory reclamation. To solve performance problems in games, Google employee Wouter van Oortmerssen developed FlatBuffers. (Note: Protocol buffers was created by Google, while FlatBuffers was created at Google.)

A few years ago, Facebook announced that the data-processing performance of its Android app had improved dramatically. Across nearly the entire app, they replaced JSON with FlatBuffers.

Like [FlatBuffers](https://github.com/google/flatbuffers) (9490 stars), [Cap'n Proto](https://github.com/capnproto/capnproto) (5527 stars), and [simple-binary-encoding](https://github.com/real-logic/simple-binary-encoding) (1351 stars), it supports “zero-copy” deserialization. During serialization, no temporary objects are created and no extra memory allocations are required. Accessing serialized data also does not require first copying it into a separate region of memory. This makes accessing data in these formats much faster than formats that require such work, such as JSON, CSV, and protobuf.

FlatBuffers is indeed quite similar to Protocol Buffers. The main difference is that FlatBuffers does not need to be parsed/unpacked before accessing data. The amount of code for the two is also in the same order of magnitude. However, Protocol Buffers has neither optional text import/export functionality nor the `union` language feature, both of which FlatBuffers provides.

FlatBuffers focuses on mobile hardware, where memory capacity and memory bandwidth are more constrained than on desktop hardware, as well as applications with the highest performance requirements: games.

## 3. FlatBuffers Adoption

After all this, readers may wonder: are many people actually using FlatBuffers? Google’s official page mentions three well-known apps and one framework that use it.

BobbleApp, India’s leading sticker app. After BobbleApp adopted FlatBuffers, the app’s performance improved significantly.

Facebook uses FlatBuffers in its Android app for client-server communication. They wrote an article, [“Improving Facebook's performance on Android with FlatBuffers”](https://code.facebook.com/posts/872547912839369/improving-facebook-s-performance-on-android-with-flatbuffers/), describing how FlatBuffers speeds up content loading.

Google’s Fun Propulsion Labs makes extensive use of FlatBuffers across all of its libraries and games.

Cocos2d-X, a leading open-source mobile game engine, uses FlatBuffers to serialize all game data.

As you can see, FlatBuffers is widely used in game-related apps.

## 4. Defining a .fbs Schema File

<p align='center'>
<img src='https://img.halfrost.com/Blog/ArticleImage/86_1.png'>
</p>

Writing a schema file allows you to define the data structures you want to serialize. Fields can be scalar types (integers/floating-point numbers of all sizes), strings, arrays of any type, references to another object, or a set of possible objects (`Union`). Fields can be optional or have default values, so they do not need to be present in every object instance.

For example:
```schema
// example IDL file

namespace MyGame;

attribute "priority";

enum Color : byte { Red = 1, Green, Blue }

union Any { Monster, Weapon, Pickup }

struct Vec3 {
  x:float;
  y:float;
  z:float;
}

table Monster {
  pos:Vec3;
  mana:short = 150;
  hp:short = 100;
  name:string;
  friendly:bool = false (deprecated, priority: 1);
  inventory:[ubyte];
  color:Color = Blue;
  test:Any;
}

root_type Monster;
```
The above is the syntax of the schema language. A schema is also known as an IDL (Interface Definition Language), and the code is very similar to languages in the C family.

In a FlatBuffers schema file, there are two very important concepts: struct and table.

### 1. Table

A Table is the primary way to define objects in FlatBuffers. It consists of a name (Monster here) and a list of fields. Each field has a name, a type, and an optional default value (if omitted, it defaults to 0 / NULL).

**Each field in a Table is optional**: it does not have to appear in the wire representation, and fields can be omitted independently for each individual object. Therefore, you can flexibly add fields without worrying about data bloat. This design is also FlatBuffers’ forward- and backward-compatibility mechanism.

Assume the current schema is as follows:
```schema
table { a:int; b:int; }
```
Now suppose you want to make changes to this schema.

There are a few points to keep in mind:

### Adding fields

New fields can only be added at the end of the table definition. Old data will still be read correctly, and default values will be provided for you when it is read. Old code will simply ignore the new fields. If you want the flexibility to use fields in any order in the schema, you can manually assign ids (much like Protocol Buffers). See the [id attribute](https://github.com/halfrost/Halfrost-Field/blob/master/contents-en/Protocol/FlatBuffers-schema.md#9-attributes) below.

Example:
```schema
table { a:int; b:int; c:int; }
```
This works. The old schema will ignore the presence of the new field `c` when reading the new data structure. The new schema will read the old data and obtain the default value of `c` (in this case, `0`, because it is not specified).
```schema
table { c:int a:int; b:int; }
```
Adding new fields at the beginning is not allowed, because it makes the old and new versions of the schema incompatible. When old code reads new data, what it reads as the new field c is actually the old field a. When new code reads old data, what it reads as the old field a is actually the old field b.
```schema
table { c:int (id: 2); a:int (id: 0); b:int (id: 1); }
```
This is feasible. If your intent is to order/group the semantics in a meaningful way, you can do so with explicit identifier assignments. Once ids are introduced, the order of fields in the table no longer matters. The new and old schemas are fully compatible as long as we preserve the id sequence.


### Removing Fields

You cannot remove fields that are no longer used from the schema, but you can simply stop writing them into the data; writing and deleting fields have almost the same effect. In addition, you can mark them as deprecated, as shown in the example above. Marked fields will no longer generate C++ accessors, thereby forcing the field to no longer be used. (Be careful: this may break code!).
```schema
table { b:int; }
```
This approach to deleting fields is not feasible. We can only remove a field by deprecating it, regardless of whether an explicit ID identifier is used.
```schema
table { a:int (deprecated); b:int; }
```
The approach above is also acceptable. The old schema will get the default value of a when reading the new data structure, because it does not exist. The new schema code can neither read nor write a (existing code that attempts to do so will result in a compilation error), but it can still read old data (it will ignore that field).


### Changing Fields

Field names and table names can be changed; if your code continues to work correctly, then you can change them as well.
```schema
table { a:uint; b:uint; }
```
Directly changing the field’s type may or may not work. It only works when the type change preserves the same size. If the old data does not contain any negative values, this is safe; if it does, this change will cause problems.
```schema
table { a:int = 1; b:int = 2; }
```
This change is not feasible. Any existing data written with values of 0 will no longer be written to the buffer and will instead rely on the recreated default values. Those values will now appear as 1 and 2. This may not cause issues in some cases, but it must be handled carefully.
```schema
table { aa:int; bb:int; }
```
With the modification approach above, issues may arise after renaming the original variable. Since the field has already been renamed, this will break all code (and JSON files) that use this version of the schema, which is incompatible with the actual binary buffer.


Tables are the cornerstone of FlatBuffers, because data-structure evolution is essential for most applications that need serialization. In general, handling data-structure changes can be done transparently during parsing in most serialization solutions. However, a FlatBuffer is not parsed before it is accessed.

To address data-structure evolution, tables access fields indirectly through a vtable. Each table carries a vtable (which can be shared among multiple tables with the same layout) and contains information about the fields stored in this particular vtable instance for this type. The vtable may also indicate that a field is absent (because this FlatBuffer was written using an older version of the software, simply because the information is not required for this instance, or because it is considered deprecated), in which case the default value is returned.

The memory overhead of tables is small (because vtables are small and shared), and the access cost is also small (an indirection), but they provide a great deal of flexibility. A table may even use less memory than an equivalent struct, because fields do not need to be stored in the buffer when they are equal to their default values.


### 2. Structs

Structs are very similar to tables, except that no fields in a struct are optional (so there are no default values), and fields may not be added or deprecated. A struct may contain only scalars or other structs. If you are certain that no future changes will be made (as is quite obvious in the Vec3 example), use it for simple objects. Structs use less memory than tables and are faster to access (they are always stored inline in their parent object and do not use a vtable).

Structs do not provide forward/backward compatibility, but they have a smaller memory footprint. Storing very small objects that are unlikely to change (such as coordinate pairs or RGBA colors) as structs is very useful.

### 3. Types

The scalar types supported by FlatBuffers are as follows:

- 8 bit: byte (int8), ubyte (uint8), bool  
- 16 bit: short (int16), ushort (uint16)  
- 32 bit: int (int32), uint (uint32), float (float32)  
- 64 bit: long (int64), ulong (uint64), double (float64)  

The names in parentheses correspond to type aliases.

The non-scalar types supported by FlatBuffers are as follows:

- Arrays of any type. However, nested arrays are not supported; you can replace nested arrays by defining an array inside a table.
- UTF-8 and 7-bit ASCII strings. Encoded strings in other formats, or binary data, should be represented with [byte] or [ubyte] instead.
- Tables, structs, enums, and unions

Fields of scalar types have default values. Fields of non-scalar types (string/vector/table) default to NULL if they have no value.

Once a type has been declared, **avoid changing its type whenever possible; once changed, errors are very likely to occur**. As mentioned above, if you change int to uint and the data contains negative numbers, errors will occur.

### 4. Enums


An enum defines a series of named constants. Each named constant can be assigned a fixed value, or it can default to one greater than the previous value. The default first value is 0. As shown in the enum declaration in the example above, use : (byte in the example above) to specify the enum’s underlying integer type, and then determine the type of each field declared with this enum type.

In general, you should only add enum values and should not delete enum values (there is no notion of deprecating enum values). **This requires developer code to handle forward compatibility by handling unknown enum values itself**.

### 5. Unions

**This is a type that Protocol buffers does not yet support**.

A union is a concept from the C language: a union can hold multiple types, all sharing the same memory region.

In FlatBuffers, however, unions share many properties with enums, but instead of new names for constants, they use the names of tables. You can declare a union field that can contain a reference to any of these types, **meaning this memory region can be used by only one of those types**. In addition, a hidden field with the suffix `_type` is generated; this field contains the corresponding enum value, allowing you to know at runtime which type to cast it to.

**A union is similar to an enum, but a union contains tables, while an enum contains scalars or structs.**

Unions are a good way to send multiple message types in a single FlatBuffer. Note that because a union field is actually two fields (there is one hidden field), it must always be part of a table; it cannot itself be the root of a FlatBuffer.

If you need to distinguish between different FlatBuffers in a more open-ended way, such as files, see the [file identification feature](https://github.com/halfrost/Halfrost-Field/blob/master/contents-en/Protocol/FlatBuffers-schema.md#7-file-identification-and-extension) below.

Finally, there is an experimental feature supported only in the C++ implementation: as in the example above, [Any] \(an array of unions) is added as a type to the Monster table definition.


### 6. Root type

This declares what you consider to be the root table (or struct) of the serialized data. This is especially important for parsing JSON data that does not contain object type information.


### 7. File identification and extension

Typically, a FlatBuffer binary buffer is not self-describing; that is, you need to know its schema in order to parse the data correctly. But if you want to use a FlatBuffer as a file format, it is convenient to have a “magic number” there, as most file formats do, so you can perform a full check to see whether you are reading the file type you expect.

Although FlatBuffer allows developers to prepend their own file header to a FlatBuffer, FlatBuffers has a built-in method that lets the identifier take minimal space while still keeping the FlatBuffer compatible with FlatBuffers that do not have such an identifier.

The way to declare a file format is similar to root\_type:
```schema
file_identifier "MYFI";
```
**The identifier must be exactly 4 characters. These 4 characters will be bytes [4,7] at the end of the buffer**.

For any schema with such an identifier, `flatc` will automatically add the identifier to any binary files it generates (with `-b`), and generated calls such as `FinishMonsterBuffer` will also add the identifier. If you have specified an identifier and want to generate a buffer without one, you can do so by explicitly calling `FlatBufferBuilder::Finish` directly.

After loading the buffer data, you can use calls such as `MonsterBufferHasIdentifier` to check whether the identifier is present.

Adding an identifier to a file is a best practice. If you simply want to send one of a set of possible messages over the network, it is better to use a Union.

By default, `flatc` outputs binary files as `.bin`. This declaration in the schema changes that to whatever you want:
```schema
file_extension "ext";
```

### 8. RPC interface declarations

RPC declares a set of functions that take a FlatBuffer as input (request) and return a FlatBuffer as the response (both of which must be of the table type):
```schema
rpc_service MonsterStorage {
  Store(Monster):StoreResponse;
  Retrieve(MonsterId):Monster;
}
```
The generated code and the way it is used depend on the language and RPC system in use. By adding the `--grpc` compiler option, the code generator provides preliminary support for GRPC.


### 9. Attributes


Attributes can be attached to field declarations, either after the field or after the name of a table/struct/enum/union. These attributes may or may not have values.

Some Attributes are recognized only by the compiler, such as deprecated. Users can also define their own Attributes, but they must be declared in advance. Once declared, they can be queried when parsing the schema at runtime. This is very useful for developing your own code compiler/generator, or for adding special information (help text, and so on) to your own FlatBuffers tooling.

The latest version currently recognizes 11 kinds of Attributes.

- `id:n` (on a table field)  
id means setting the identifier of a field to n. Once this id identifier is enabled, all fields must use id identifiers, and the ids must be consecutive numbers starting from 0. Special attention is required for Union: because a Union consists of 2 fields, and the hidden field comes before the union field. (Suppose the id of the field before the union reaches 6; then the union will occupy id numbers 7 and 8, where 7 is the hidden field and 8 is the union field.) After adding id identifiers, the relative ordering of fields inside the schema no longer matters. The id used by a new field must be the next immediately available id (ids cannot be skipped; they must be consecutive).
- `deprecated` (on a field)  
deprecated means accessors will no longer be generated for this field, and code should stop using this data. Old data may still contain this field, but new code can no longer access it. Note that if you deprecate a previously required field, old code may fail to validate new data (when using the optional verifier).
- `required` (on a non-scalar table field)  
required means this field cannot be omitted. By default, all fields are optional, meaning they can be omitted. This is desirable because it helps with forward/backward compatibility and data-structure flexibility. It is also a burden on reader code, because for non-scalar fields it requires you to check for NULL and take appropriate action. By specifying a required field, code that builds FlatBuffers can be forced to ensure that this field is initialized, so reader code can access it directly without checking for NULL. If builder code does not initialize this field, it will get an assertion indicating that a required field is missing. Note that if you add this attribute to an existing field, it is valid only if existing data always contains this field / existing code always writes this field.
- `force_align: size` (on a struct)  
force\_align means forcing this struct to have a higher alignment than its natural alignment. If the buffer is created with force\_align declared, all structs inside it will be forcibly aligned. (For buffers accessed directly in FlatBufferBuilder, this is not necessarily the case.)  
- `bit_flags` (on an enum)  
bit\_flags means the values of this field represent bits. This means any value N specified in the schema will eventually represent 1 << N, or, if no values are explicitly specified, the default sequence will be 1, 2, 4, 8, ...
- `nested_flatbuffer: "table_name"` (on a field)  
nested\_flatbuffer means this field (which must be an array of ubyte) contains nested flatbuffer data, whose root type is given by table\_name. The generated code will generate a convenient accessor for the nested FlatBuffer.
- `flexbuffer` (on a field)  
flexbuffer means this field (which must be an array of ubyte) contains flexbuffer data. The generated code will create a convenient accessor for the FlexBuffer root.
- `key` (on a field)  
The key field is used in the current table as the key when sorting an array of the type it belongs to. It can be used for in-place lookup via binary search.
- `hash` (on a field)  
This is an unsigned 32/64-bit integer field. During JSON parsing, its value is allowed to be a string, and then it is stored as its hash. The attribute value is the hashing algorithm to use, namely one of fnv1\_32, fnv1\_64, fnv1a\_32, fnv1a\_64.  
- `original_order` (on a table)  
Because elements in a table do not need to be stored in any particular order, they are usually sorted by size to optimize space. original\_order prevents this from happening. In general, there should be no reason to use this flag.
- 'native_*'  
Several attributes have been added to support the [C++ object-based API](https://google.github.io/flatbuffers/flatbuffers_guide_use_cpp.html#flatbuffers_cpp_object_based_api), all prefixed with “native\_”. For details, click the [link](https://google.github.io/flatbuffers/flatbuffers_guide_use_cpp.html#flatbuffers_cpp_object_based_api) to view the supported documentation: `native_inline`, `native_default`, `native_custom_alloc`, `native_type`, `native_include: "path"`。

### 10. Design Recommendations

FlatBuffers is an efficient data format, but to achieve efficiency, you need an efficient schema. There are often multiple ways to represent data with completely different size characteristics.

Because FlatBuffers is flexible and extensible, it is very common to represent any type of data as a dictionary (as in JSON). Although this can be simulated in FlatBuffers (as an array of tables with keys and values), doing so is an inefficient approach for a strongly typed system like FlatBuffers, and it leads to relatively large binaries. In most systems, a FlatBuffer table is more flexible than classes/structs, because a table remains very efficient when it has a very large number of fields but only a few are actually used. Therefore, data should be organized as tables as much as possible.

Similarly, whenever possible, prefer enums over strings.

FlatBuffers has no concept of inheritance, so the way to represent a group of related data structures is a union. However, unions do have a cost. Another efficient approach is to create a table. If these data structures have many similar or shareable fields, then using a single table is very efficient. Simply include all fields of all data structures in this table. The reason this is efficient is that optional fields are very cheap and have low overhead.

FlatBuffers supports all integer widths by default, so choose the smallest size you need rather than defaulting to int/long.

Consider sharing common data by using a string or table in the buffer. This improves efficiency, so splitting duplicated data into shared data structures + private data structures is very worthwhile.


## V. FlatBuffers JSON Parsing


<p align='center'>
<img src='https://img.halfrost.com/Blog/ArticleImage/86_2.png'>
</p>


FlatBuffers supports parsing JSON into its own format. In other words, the parser that parses schemas can also parse JSON objects that conform to the schema rules. Therefore, unlike other JSON parsers, this parser is strongly typed, and the parsing result is only FlatBuffers. For the exact approach, refer to the flatc documentation and the corresponding C++ FlatBuffers documentation to see how to parse JSON into FlatBuffers at runtime.

To parse JSON, in addition to defining a schema, the FlatBuffers parser has the following changes:

- It accepts field names both with and without quotes, just as many JSON parsers already do. It can also output them without quotes, but you can use the `strict_json` flag to output them with quotes.
- If a field has an enum type, the parser recognizes symbolic enum values (with or without quotes) rather than numbers, for example field: EnumVal. If a field is an integer type, you can still use symbolic names, but these values need to be prefixed with their type and enclosed in quotes: field: “Enum.EnumVal”. For enums that represent flags, you can insert spaces between multiple strings or use dot syntax. For example: field: “EnumVal1 EnumVal2” or field: “Enum.EnumVal1 Enum.EnumVal2”.
- For unions, these need to be specified using two fields, just as when serializing from code. For example, for a field foo, you must add a `foo_type：FooOne` before the foo field, where FooOne is a table that can be used outside the union.
- If the value of a field is null (for example, field: null), it means this field has the default value (which has the same effect as not specifying the field at all).
- The parser has several built-in conversion functions, so where you would otherwise write 3.14159, you can use the rad(180) function instead. The currently supported functions are: rad, deg, cos, sin, tan, acos, asin, atan.

When parsing JSON, the parser recognizes the following escape codes in strings:

`\n` - Newline.   
`\t` - Tab.   
`\r` - Carriage return.   
`\b` - Backspace.   
`\f` - Form feed.   
`\“` - Double quotation mark.   
`\\` - Backslash.   
`\/` - Forward slash.  
`\uXXXX` - 16-bit Unicode, converted to the equivalent UTF-8 representation.   
`\xXX` - 8-bit binary hexadecimal number XX. This is the only place that is not part of the JSON specification (see [http://json.org/](http://json.org/)), but it is needed to encode arbitrary binary data in strings as text and recover it without losing information (for example, byte 0xFF cannot be represented as standard JSON).

When generating JSON back from the binary representation, it will generate these escape codes again.


## VI. FlatBuffers Naming Conventions

Identifiers in a schema are meant to be translated into many different programming languages, so changing the schema’s coding style to match the style used by the current project language is the wrong approach. The schema’s coding style should be more general.

- Table, struct, enum and rpc names (types) use UpperCamelCase.
- Table and struct field names use snake_case. This allows generated methods to automatically use lowerCamelCase code.
- Enum values use UpperCamelCase.
- namespaces use UpperCamelCase.

There are also 2 recommendations about formatting:

- Braces: on the same line as the start of the declaration.
- Spacing: indent by 2 spaces. No spaces around `:`, and one space on each side of `=`.

## VII. Some FlatBuffers "Pitfalls"

<p align='center'>
<img src='../images/flatbuffer_attention.gif'>
</p>


Most serializable formats (such as JSON or Protocol Buffers) are very explicit about whether a field exists in an object, and this can be used as “extra” information. 

But in FlatBuffers, apart from scalar values, this applies to everything else as well. By default, FlatBuffers does not write fields whose values are equal to their defaults (for scalars), which can save a lot of space. However, this also means that testing whether a field “exists” is somewhat meaningless, because it does not tell you whether the field was set by calling the add_field method, unless you care about the information that it has a non-default value. Default values are not written into the buffer.


The mutable FlatBufferBuilder implements a method named force\_defaults, which can avoid this behavior, because fields are written even if they are equal to their default values. You can then use IsFieldPresent to query whether a field exists in the buffer. 

Another approach is to wrap scalar fields in a struct. That way, if it does not exist, it will return null. What makes this approach powerful is that structs do not occupy more space than the scalars they represent.


## VIII. Conclusion 


After reading this article on FlatBuffers encoding principles, readers should understand the following points:

Compared with protocol buffers, FlatBuffers data-structure definition files have the following functional “improvements”:

- Deprecated fields do not require manually assigning field IDs. In `.proto`, when extending an object, you need to find a free slot among the numbers (because protocol buffers have a more compact representation, smaller numbers must be chosen). Besides this inconvenience, it also makes deleting fields problematic: if you keep them, semantically it is not very obvious that the field can no longer be read or written, and keeping them also generates accessors. If you delete them, there is a risk of serious bugs, because if someone reuses these IDs, old data may be read incorrectly, causing data corruption.
- FlatBuffers distinguishes between table and struct. All table fields are optional, and all struct fields are required.
- FlatBuffers has a native array type rather than repeated. This gives you a length without having to collect all items, provides a more compact representation for scalars, and ensures adjacency.
- FlatBuffers has a union type, which protocol buffers do not have. A union can replace many optional fields, which also saves the time required to check each field one by one.
- FlatBuffers can define default values for all scalars without having to handle their optionality on every access, and default values are not stored in the buffer, so there is no need to worry about space.
- A parser can uniformly handle schemas and data definitions (and is compatible with JSON). protocol buffers is not compatible with JSON. The flatc compiler for FlatBuffers also supports more powerful options. For the specific list of available parameters, see [this document](https://google.github.io/flatbuffers/flatbuffers_guide_using_schema_compiler.html)
- schema extends some Attributes that protocol buffers does not have.

Apart from functional differences, there are also some subtle differences in schema syntax:

- To define an object, protocol buffers uses message, while FlatBuffers uses table
- ID numbering in protocol buffers starts from 1 by default, while FlatBuffers starts from 0 by default.


For all schema syntax, refer to [this document](https://google.github.io/flatbuffers/flatbuffers_grammar.html)
