## Starting Point

Blog page optimization starts with Chrome’s Performance panel.

<p align='center'>
<img src='https://img.halfrost.com/Blog/ArticleImage/94_1_.png'>
</p>

As you can see from the figure above, the page is blank for the first 550 ms, with the time spent loading JS. This segment consists of the Disqus script and the Google Analytics script.

My optimization principle is: **pre-load what must be loaded, and lazy-load what does not have to be loaded and is not currently visible on the screen**.

The optimization idea is simple: before DOM rendering, remove the long yellow rectangle in the figure above. That time is spent loading JS, which blocks page rendering. When rendering the page, I do not load the Disqus script; only when it is needed on a blog post page—when the button is clicked—will the script be lazy-loaded.

As for the Google Analytics script, using the latest asynchronous version of gtag and adding the async flag is enough to solve the issue. Of course, if you want to make this analytics code fully asynchronous, you need to make some server-side changes.

For reference, see these two articles: [“Optimizing Asynchronous Loading of Google Analytics to Improve Your Website Speed”](https://imiku.me/2017/07/14/916.html) and [“Using Google Analytics on the Server Side”](https://blog.alphatr.com/google-analytics-on-server.html).

If you also have the Baidu Analytics script, move the code generated by Baidu Analytics into the website and set type to a non-MIME/type value such as test. This way, the browser will not parse this script, while Baidu Analytics can still pass validation:
```javascript
<script type="text/test">
    var _hmt = _hmt || [];
    (function() {
      var hm = document.createElement("script");
      hm.src = "https://hm.baidu.com/hm.js?XXXX";
      var s = document.getElementsByTagName("script")[0];
      s.parentNode.insertBefore(hm, s);
    })();
    </script>
```
For Baidu Analytics, you can refer to this article: [“Deferred Asynchronous Loading of Analytics Code”](http://blog.angular.in/analyticsdai-ma-yan-chi-yi-bu-jia-zai/).


Once the two items above are handled properly, you will get the result shown below:

<p align='center'>
<img src='https://img.halfrost.com/Blog/ArticleImage/94_14_.png'>
</p>

The effect is very obvious: the homepage’s First Meaningful Paint time (FMP, the time it takes for the main content to appear on the page) is now under 100 ms.

## Benchmarking

Next up is Chrome Audits. The author’s first benchmark result was not great. The lower scores were in Performance and Accessibility.

<p align='center'>
<img src='https://img.halfrost.com/Blog/ArticleImage/94_2_.png'>
</p>

For Performance, the following metrics are evaluated:

- First Meaningful Paint (FMP, the time it takes for the main content to appear on the page),
- Time to meaningful rendering (the time it takes for the most important part of the page to finish rendering),
- Time to Interactive (TTI, when the page layout has stabilized, key page fonts are visible, and the main thread can sufficiently handle user input — the basic time marker is that the user can click and interact with the UI),
- Input responsiveness, the time required for the interface to respond to user actions,
- Speed Index, which measures how quickly page content is populated. The lower the score, the better,

A common definition of smoothness is:

Keep response time within 100ms and the frame rate at 60 frames/second. Speed Index < 1250, TTI < 5s on 3G, Critical file size budget < 170Kb（SpeedIndex < 1250, TTI < 5s on 3G, Critical file size budget < 170Kb）

The first 14–15kb of HTML is the most critical payload chunk — and the only part that can be delivered within the first round trip (what you get within 1 second under a 400 ms round-trip latency budget). In general, to achieve the goals above, we must operate within the critical file size budget. The maximum budget is 170 Kb compressed (0.8–1MB decompressed), which already consumes as much as 1s (depending on the resource type) for parsing and compilation. Slightly exceeding this value is acceptable, but these numbers should be kept as low as possible.

Because the homepage of this blog uses a large number of images, image optimization is the major focus. In the screenshot above, Chrome gives three optimization suggestions for images:

- Use JPEG 2000, JPEG XR, or WebP instead of PNG and JPEG whenever possible, so image sizes are smaller.
- The image dimensions fetched from the server should preferably already be the scaled dimensions; do not download overly large images.
- Offscreen rendering. Lazy-load images outside the viewport.

- Reduce render-blocking stylesheets
- Reduce unused CSS styles
- Reduce render-blocking JS scripts

For Accessibility, the following metrics are easy to overlook:

- Elements should use the correct Attributes; for example, add the `[alt]` Attribute to `<image>`.
- Elements should have a Discernible name; for example, add a Discernible name to `<a>`.
- Form elements are missing associated labels.
- Background and foreground colors should have sufficient color differentiation: sufficient contrast ratio.

For the last point, it is recommended to install the aXe extension plugin in Chrome and use it during debugging to inspect color contrast.

## Optimization

The focus of optimization is, of course, images. There are three main aspects: use WebP for the image format, scale images to an appropriate size, and lazy-load images.

Here is a brief explanation of how lazy-load works: in the viewport, you can obtain the percentage of an element that is currently within the visible window. Based on this ratio, we know which element is about to be loaded next.

<p align='center'>
<img src='https://img.halfrost.com/Blog/ArticleImage/94_15_.png'>
</p>

You can add a `data-url` attribute to the image tag to store the image’s real URL.
```html
<img class="lazy" data-url="{{img_url feature_image}}" alt="">
```
When scrolling reaches the point where an image needs to be displayed, perform some processing, and assign the final, actual image URL to the `src` attribute. The browser will then load the specified image.

<p align='center'>
<img src='https://img.halfrost.com/Blog/ArticleImage/94_16_.png'>
</p>

For a more detailed explanation, see Google’s official article, [“IntersectionObserver’s Coming into View”](https://developers.google.com/web/updates/2016/04/intersectionobserver).

Converting images to WebP requires using the API provided by Qiniu. If you use another CDN, you can also check whether they provide a similar API. If they do, that makes things much easier.

Qiniu provides a set of APIs for basic image processing. See this official document: [“Qiniu Basic Image Processing API”](https://developer.qiniu.com/dora/manual/1279/basic-processing-images-imageview2).

The basic-processing API above also includes APIs for resizing images, so when fetching images from the CDN, we can resize them to an appropriate size and convert them to WebP in one step. Then the final image URL can be assigned directly to `src`.

One thing to note, however, is that browsers on Apple platforms such as Safari and iOS do not support WebP. If you force them to load WebP, the image quality degrades severely, the resolution is poor, and the rendered result will look very blurry. So before loading WebP, you need to detect whether it is supported. If it is not supported, you should fall back to JPEG 2000 or JPEG XR.

Other optimizations include optimizing JS and CSS files. Before publishing to production, always remember to minify these asset files. After minification, their size can become much smaller, reducing the time spent on network transfer. If JS files can be split into chunks and loaded on demand, the results will be even better.

The key is to inspect every JavaScript dependency. Tools such as webpack-bundle-analyzer, Source Map Explorer, and Bundle Buddy can help you do this. Measure JavaScript parse and compile time. Etsy’s DeviceTiming is a small tool that lets your JavaScript measure parse and execution time on any device or browser. Importantly, while file size matters, it is not the most important factor. Parse and compile time do not increase linearly with script size.

For optimizing JS files, Webpack is the first recommendation.

Code-splitting is a Webpack feature that breaks your code into “chunks” loaded on demand. Not all JavaScript must be downloaded, parsed, and compiled. Once you identify split points in your code, Webpack handles the dependencies and output files. When the application sends requests, this largely ensures that the initial download is small enough and that loading happens on demand. Also, consider using preload-webpack-plugin to obtain the paths produced by code splitting, and then use `<link rel="preload">` or `<link rel="prefetch">` to hint to the browser to preload them.

If you cannot use Webpack, Rollup’s output is generally better than Browserify’s. When using Rollup, I recommend looking into Rollupify, which can transform ECMAScript 2015 modules into one large CommonJS module—because small modules can have surprisingly high performance overhead (depending on the choice of bundler and module loading system).

To ensure the browser starts rendering the page as soon as possible, you typically collect all the CSS needed to render the first visible portion of the page (called “critical CSS” or “above-the-fold CSS”) and inline it into the page’s `<head>`, reducing round trips. Because the amount of data that can be exchanged during the slow-start phase is limited, the budget for critical CSS is roughly 14 KB.

### 1. PWA

<p align='center'>
<img src='https://img.halfrost.com/Blog/ArticleImage/94_17_.png'>
</p>


No network performance optimization can beat the local cache on the user’s machine. If your site runs over HTTPS, you can consider implementing a PWA. The purpose is to use a service worker to cache static assets and store offline resources (even offline pages). It also teaches you how to retrieve data from the user’s device, which means you no longer need to request previously fetched data over the network.

PWA provides a particularly large improvement the second time a user opens the page, because the assets are all loaded locally.

I spent some effort on PWA and recommend reading this official article: [“PWA Web App Manifest”](https://developers.google.com/web/fundamentals/web-app-manifest/#_5).

Because of the nature of a blog, pages do not change much, so many assets can be cached. My PWA caching strategy is to cache JS and CSS framework files, fonts, the Disqus framework, images, and the Baidu and Google analytics frameworks. This way, when users load the site for the second time, 99% of requests hit the cache, greatly improving load speed.

There is a well-written article about Service Workers that I recommend: [“Using Service Workers”](https://developer.mozilla.org/zh-CN/docs/Web/API/Service_Worker_API/Using_Service_Workers).

### 2. HTTP/2

This also requires Nginx support.
```nginx
server {
    listen 443 ssl http2;
}
```
The improvements from HTTP/2 are substantial. I’ll analyze this in a later article in the series, so I won’t cover it in this optimization article.

If HTTP/2 is still not enabled after applying the configuration above, you can refer to this article: [Fixing the Issue Where Nginx HTTP/2 Configuration Does Not Take Effect and Google Chrome Still Uses the HTTP/1.1 Protocol](https://zhangge.net/5114.html)

### 3. Enable gZip in Nginx

gZip can effectively reduce network transfer overhead. After it is enabled, it will consume a small amount of server CPU, but it can help improve frontend page performance. Here is the Nginx configuration from my own server:
```nginx
http {
    include            mime.types;
    default_type       application/octet-stream;

    charset            UTF-8;

    sendfile           on;
    tcp_nopush         on;
    tcp_nodelay        on;

    keepalive_timeout  60;

    #... ...#

    gzip               on;
    gzip_vary          on;

    gzip_comp_level    6;
    gzip_buffers       16 8k;

    gzip_min_length    1000;
    gzip_proxied       any;
    gzip_disable       "msie6";

    gzip_http_version  1.0;

    gzip_types         text/plain text/css application/json application/x-javascript text/xml application/xml application/xml+rss text/javascript application/javascript image/svg+xml;

    #... ...#

    include            /home/jerry/www/nginx_conf/*.conf;
}
```

### 4. Support TLS 1.3

There are plenty of tutorials online for this, so I won’t list specific links here.

The main steps are to first install the latest OpenSSL and Nginx, then compile OpenSSL, and then compile Nginx. When compiling Nginx, include the OpenSSL you just built and the compilation options for TLS 1.3.

To enable TLS 1.3, you can refer to these two articles: [“This Site Now Supports TLS 1.3”](https://www.cainwang.cn/tls1_3support/) and [“This Blog Now Supports TLS 1.3”](https://imququ.com/post/enable-tls-1-3.html#comment-4000795428).

### 5. Enable OCSP Stapling
```nginx
server {
    ssl_session_cache        shared:SSL:10m;
    ssl_session_timeout      60m;

    ssl_session_tickets      on;

    ssl_stapling             on;
    ssl_stapling_verify      on;
    ssl_trusted_certificate  /xxx/full_chain.crt;

    resolver                 8.8.4.4 8.8.8.8  valid=300s;
    resolver_timeout         10s;
    ... ...
}
```
The purpose of TLS session resumption is to simplify the TLS handshake. There are two approaches: Session Cache and Session Ticket. Both store the Session from a previous handshake for use by subsequent connections. The difference is that Cache is stored on the server and consumes server-side resources, while Ticket is stored on the client and does not consume server-side resources. In addition, mainstream browsers currently all support Session Cache, while support for Session Ticket is more limited.

The first few lines beginning with `ssl_stapling` are used to configure the OCSP stapling policy. A browser may verify certificate validity online when establishing a TLS connection, which can block the TLS handshake and slow down overall performance. OCSP stapling is an optimization: it allows the server to include the certificate authority’s OCSP (Online Certificate Status Protocol) response in the certificate chain, letting the browser skip the online query. Fetching OCSP from the server is faster on the one hand (because servers generally have better network conditions), and easier to cache effectively on the other.


Let's Encrypt currently supports ECC certificates. After obtaining one, remember to enable OCSP Stapling as well. For the specific steps, refer to this article: [“Let's Encrypt: A Free and Easy-to-Use HTTPS Certificate”](https://imququ.com/post/letsencrypt-certificate.html)

After configuration, you need to verify whether OCSP Stapling is enabled. You can use the following command:
```nginx
$ cd /var/www/ghost/ssl/

$ openssl ocsp -CAfile full_chained.pem -issuer intermediate.pem -cert chained.pem -no_nonce -text -url http://ocsp.int-x3.letsencrypt.org -header "HOST" "ocsp.int-x3.letsencrypt.org"

```

### 6. Support QUIC

This blog now supports QUIC, but I haven’t noticed much of a performance improvement. Also, currently only Chrome supports QUIC, and the chance of it being enabled is very low; most traffic still goes over HTTP/2.

To support QUIC, you need to deploy caddy, a Go project, on the server. Since we already have Nginx occupying port 443, we need to set up port mapping, which can be done with Docker. In addition, the Nginx configuration also needs to add a header parameter:
```nginx
add_header alt-svc 'quic=":443"; ma=2592000; v="39"';
```

### 7. Enable HSTS

Enabling HSTS requires the following conditions to be met first:

- Have a valid certificate (if using a SHA-1 certificate, its expiration date must be earlier than 2016);
- Redirect all HTTP traffic to HTTPS;
- Ensure that HTTPS is enabled for all subdomains;
- Output the HSTS response header:  
  max-age must not be less than 18 weeks (10886400 seconds);  
  the includeSubdomains parameter must be specified;  
  the preload parameter must be specified;  

Then Nginx needs to add the relevant security policy.
```nginx

add_header  Strict-Transport-Security  "max-age=31536000";
add_header  X-Frame-Options  deny;
add_header  X-Content-Type-Options  nosniff;
add_header  Content-Security-Policy  "default-src 'self'; script-src 'self' 'unsafe-inline' 'unsafe-eval' https://a.disquscdn.com; img-src 'self' data: https://www.google-analytics.com; style-src 'self' 'unsafe-inline'; frame-src https://disqus.com";

ssl_certificate      /home/ssl/server.crt;
ssl_certificate_key  /home/ssl/server.key;
ssl_dhparam          /home/ssl/dhparams.pem;

ssl_ciphers          ECDHE-ECDSA-CHACHA20-POLY1305:ECDHE-RSA-CHACHA20-POLY1305:ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES128-GCM-SHA256:ECDHE-ECDSA-AES256-GCM-SHA384:ECDHE-RSA-AES256-GCM-SHA384:ECDHE-ECDSA-AES128-SHA:ECDHE-RSA-AES128-SHA:ECDHE-ECDSA-AES256-SHA384:ECDHE-RSA-AES256-SHA384:ECDHE-ECDSA-AES256-SHA:ECDHE-RSA-AES256-SHA:ECDHE-ECDSA-AES128-SHA256:ECDHE-RSA-AES128-SHA256:AES128-GCM-SHA256:AES256-GCM-SHA384:DES-CBC3-SHA;

ssl_prefer_server_ciphers  on;

ssl_protocols        TLSv1 TLSv1.1 TLSv1.2;
```
