# Step-by-Step Guide to Configuring Multiple Environment Variables for an iOS App


![](https://img.halfrost.com/Blog/ArticleTitleImage/19_0_.jpg)


#### Preface  
When it comes to multiple environments, I believe most companies now have at least 2–3 app environments, such as a Test environment, a UAT (User Acceptance Test) environment, a Release environment, and so on. When developers need to produce multiple builds, the common approach is usually to modify the environment variables directly in the code, then run Archive to package the app. Of course, this approach works, but it is neither elegant nor efficient. If Jenkins has been set up ([setup tutorial](http://www.jianshu.com/p/41ecb06ae95f)), we can use it to package the app in a more elegant way. If we use Jenkins for packaging, we need to configure multiple environment variables for the app. After that, Jenkins can automatically integrate builds for different environments. Next, let’s discuss two common approaches.

####Table of Contents
- 1. Configure multiple environments using Build Configuration
- 2. Configure multiple environments using [xcconfig](https://developer.apple.com/library/ios/recipes/xcode_help-project_editor/Articles/BasingBuildConfigurationsonConfigurationFiles.html) files
- 3. Configure multiple environments using Targets

#### 1. Configure Multiple Environments Using Build Configuration
In the preface, we first discussed the requirement: we need to configure multiple environments, and each environment needs to be installable on a device. Build Configuration can be used to accomplish this. If you are not familiar with Build Configuration yet, you can first review the [official documentation](https://developer.apple.com/library/ios/recipes/xcode_help-project_editor/Articles/BasingBuildConfigurationsonConfigurationFiles.html), or the [newer documentation here: Build Settings Reference](https://help.apple.com/xcode/mac/8.0/#/itcaec37c2a6).

##### 1. Create a New Build Configuration
First, click the Project and find Configuration, then choose to add one. Here we add a new Configuration. By default, the system provides two: Debug and Release. Here we need to choose whether to duplicate Debug or Release. The difference between Release and Debug is that Release builds cannot be debugged, because by default some debugging-related parameters are disabled. You can see the specific differences in Build Settings. In addition, Release builds apply compiler optimizations, so the resulting package is slightly smaller than one built with Debug.

![](https://img.halfrost.com/Blog/ArticleImage/19_2.png)  


Here we choose Duplicate “Debug” Configuration, because our new environment needs to be debuggable. After adding it, there will be one additional set of Configuration. This set is essentially a collection of build parameter settings. If the project uses CocoaPods at this point, opening Configuration Set will show something like this:

![](https://img.halfrost.com/Blog/ArticleImage/19_3.png)  


In our own project, where Pod is used, opening the configuration will show the following information:  

![](https://img.halfrost.com/Blog/ArticleImage/19_4.png)  


Note: After creating the new Build Configuration, if you are using Pods, immediately run the following.
```vim  

pod install  
```
After pod installation completes, an xcconfig file is generated automatically. If you manually create this xcconfig file and copy the contents from the original pod xcconfig files for debug and release, it will not work. The xcconfig file must be generated by pod itself in order to be recognized.

After creating the Build Configuration, you also need to create the corresponding Build Configuration in Pods; otherwise the build will fail later. If you are not using pods, you can ignore this section.

Create a Build Configuration corresponding to the one previously created in the Project, as shown below.  

![](https://img.halfrost.com/Blog/ArticleImage/19_5.png)

Go back to the Project’s Build Settings and select Add User-Defined Setting.  

![](https://img.halfrost.com/Blog/ArticleImage/19_9.png) 


Here we add two new parameters. CustomAppBundleld is used so that later packaging can generate multiple separate packages. Here, three different IDs are required. It is recommended to simply append the Scheme name to the original Bundleld.

CustomProductName is the name displayed on the phone after the app is installed. You can describe it according to the corresponding environment, such as testing, UAT, and so on. As shown below.

First, modify the info.plist file.

 Name for each of these modes to the corresponding icon, as shown above.


Now that we have created these schemes, how do we package all of them as apps? Here is an official document, [Troubleshooting Application Archiving in Xcode](https://developer.apple.com/library/ios/technotes/tn2215/_index.html), which records in detail how packaging works when we usually click Archive.

Here I’ll share some of my experience after separating these environments. Always remember: configure both Debug and Release for every environment! Never assume that the production version only needs Release configured. If one day you need to debug the production version and Debug has not been configured, you will have nowhere to start. Also, never assume that the test environment only needs Debug configured. If one day you need to publish a Release package for the test environment, you will again have nowhere to start. My recommendation is to configure both Debug and Release for every environment. Even if you do not use them later, set them up in advance just in case. A reasonable setup should look like the figure below.
```markdown  
| -------------------------- |------------------|
|           Scheme           |   Configurations |  
| -------------------------- |------------------| 
|      XXXXProjectTest       |      Debug       | 
|                            |------------------|
|                            |      Release     | 
| -------------------------- |------------------|
|      XXXXProjectAppStore   |      Debug       | 
|                            |------------------|
|                            |      Release     | 
| -------------------------- |------------------|
|      XXXXProjectUAT        |      Debug       | 
|                            |------------------|
|                            |      Release     | 
| -------------------------- |------------------|

```
Be careful here to distinguish between the Scheme name and the build configuration. Selecting a Scheme is merely equivalent to selecting an environment; it does not indicate whether it is Debug or Release.

![](https://img.halfrost.com/Blog/ArticleImage/19_15.png)  


I recommend using Scheme only to configure the environment, while using the incoming Run and Archive actions to configure Debug and Release. I recommend configuring every Scheme as shown above: Run corresponds to Debug, and Archive corresponds to Release.

##### 5. Configuring and Retrieving Environment Variables

Next, let’s cover several ways to dynamically configure environment variables.

###### 1. Using the GCC Precompiled Header Parameter GCC_PREPROCESSOR_DEFINITIONS  

Go to Build Settings and find Apple LLVM Preprocessing. Here you can find **Preprocessor Macros**. This is where you can add macro definitions for environment variables as identifiers. Preprocessor Macros can define different macros in advance for different environments.

![](https://img.halfrost.com/Blog/ArticleImage/19_17.png)  

As shown above, the circled area is actually an identifier.

With these identifiers preconfigured, we can write the following code in our source.
```objectivec

#ifdef DEVELOP

#define searchURL @"http://www.baidu.com"

#define sociaURL  @"weibo.com"

#elif UAT

#define searchURL @"http://www.bing.com"

#define sociaURL  @"twitter.com"

#else

#define searchURL @"http://www.google.com"

#define sociaURL  @"facebook.com"

#endif

```

###### 2. Dynamically Configure Environment Variables Using plist Files  

First, create three plist files with the same name as the configuration files for the three environments.  

![](https://img.halfrost.com/Blog/ArticleImage/19_18.png)  


The benefit of using the same name here is that it makes the code simpler: we only need to read `Configuration.plist`. If the names were different, we would have to construct the corresponding plist name for each environment before reading it.

As we all know, if we create two files with the same name in a single folder, macOS will warn us that the names are identical and will not allow us to create them. So how do we create three files with the same name? This is actually very simple: place them in three different folders. As shown below:

![](https://img.halfrost.com/Blog/ArticleImage/19_19.png)   

This is how I organized them. You can place the files however you prefer.


Next, before running the app during the build, we need to dynamically copy `Configuration.plist` into the app. This requires setting up a copy script.

![](https://img.halfrost.com/Blog/ArticleImage/19_20.png)  


Go into our Target, find **Build Phases**, create a **New Copy Files Phase**, and rename it to **Copy Configuration Files**.
```vim

echo "CONFIGURATION -> ${CONFIGURATION}"
RESOURCE_PATH=${SRCROOT}/${PRODUCT_NAME}/config/${CONFIGURATION}

BUILD_APP_DIR=${BUILT_PRODUCTS_DIR}/${PRODUCT_NAME}.app

echo "Copying all files under ${RESOURCE_PATH} to ${BUILD_APP_DIR}"
cp -v "${RESOURCE_PATH}/"* "${BUILD_APP_DIR}/"

```
This script ensures that, at compile time, one of our `Configuration.plist` files can be selected and packaged into our app.

Then, by writing code to read the information from this plist each time, we can make the configuration dynamic.
```objectivec

- (NSString *) readValueFromConfigurationFile {
    NSBundle *bundle = [NSBundle mainBundle];
    NSString *path = [bundle pathForResource:@"Configuration" ofType:@"plist"];
    NSDictionary *config = [NSDictionary dictionaryWithContentsOfFile:path];
    return config[@"serverURL"];
}

```
Here I assume the plist file has a `serverURL` string preset; this approach lets you read it out. Of course, you can also define arrays or dictionaries in the plist—just adjust the return value and key accordingly.


###### 3. Use a singleton to handle environment switching

Of course, you can also use a singleton to support environment switching. Create a singleton, then add a list to the settings menu that lists all available environments. After the user makes a selection, the singleton initializes the environment chosen by the user. Unlike the approaches above, this method switches among multiple environments within a single app. Choose whichever approach fits your needs.


 

#### II. Use files to configure multiple environments
Speaking of [xcconfig](https://developer.apple.com/library/ios/recipes/xcode_help-project_editor/Articles/BasingBuildConfigurationsonConfigurationFiles.html), Apple’s official documentation does not go into much detail. After searching online, I found another detailed unofficial document: [The Unofficial Guide to xcconfig files](http://pewpewthespells.com/blog/xcconfig_guide.html#CondVarSDK)

Before discussing xcconfig, let’s first clarify a few concepts.


##### 1. Distinguish several concepts 
First, let’s clarify the relationship among Xcode Workspace, Xcode Scheme, Xcode Project, Xcode Target, and Build Settings. Apple’s official documentation already explains the relationship among these five concepts very clearly. For details, see [Xcode Concepts](https://developer.apple.com/library/ios/featuredarticles/XcodeConcepts/Concept-Targets.html).

Here is my brief interpretation of the documentation.

**Xcode Workspace**   

> A workspace is an Xcode document that groups projects and other documents so you can work on them together. A workspace can contain any number of Xcode projects, plus any other files you want to include. In addition to organizing all the files in each Xcode project, a workspace provides implicit and explicit relationships among the included projects and their targets.  

The concept of a workspace should be familiar to everyone. It can contain multiple projects and other document files.


**Xcode Project**

>An Xcode project is a repository for all the files, resources, and information required to build one or more software products. A project contains all the elements used to build your products and maintains the relationships between those elements. It contains one or more targets, which specify how to build products. A project defines default build settings for all the targets in the project (each target can also specify its own build settings, which override the project build settings).

A project is a repository. It contains all the files and resources belonging to the project, as well as the information needed to build one or more software products. Each project contains one or more targets, and each target tells us how to produce products. A project defines the default build settings for all targets. Each target can also define its own build settings, and the target’s build settings override the project’s build settings.

This last point is important; we will use it below when configuring xcconfig.


![](https://img.halfrost.com/Blog/ArticleImage/19_21.png)  

An Xcode Project file contains the following information: references to resource files (source `.h` and `.m` files, frameworks, resource files such as plist files, bundle files, image files such as image.xcassets, as well as Interface Builder (nib) and storyboard files), groups used to organize source files in the file navigator, project-level build configurations (Debug\\Release), targets, and the executable environment used to debug or test the program.

**Xcode Target**

>A target specifies a product to build and contains the instructions for building the product from a set of files in a project or workspace. A target defines a single product; it organizes the inputs into the build system—the source files and instructions for processing those source files—required to build that product. Projects can contain one or more targets, each of which produces one product.

A target produces one, and only one, product. It organizes the files required to build that product, along with the instructions for processing those files, into the build system. Projects can contain one or more targets, and each target produces one product.


It is worth noting that the build setting parameters in each target are inherited from the project’s build settings. Once you modify any setting in a target to override the project settings, the final effective setting is the one configured in the target. A project can contain multiple targets, but at any given time only one target is active. You can use an Xcode scheme to specify which target is active.


**Build Settings**

>A build setting is a variable that contains information about how a particular aspect of a product’s build process should be performed. For example, the information in a build setting can specify which options Xcode passes to the compiler.

Build settings contain the parameter information required during the product build process. The project’s build settings apply to all targets in the entire project, while a target’s build settings override the project’s build settings; the overridden configuration is determined by the target.

A build configuration specifies a set of build settings used to generate the product of a target. For example, Debug and Release are both build configurations.


**Xcode Scheme**

>An Xcode scheme defines a collection of targets to build, a configuration to use when building, and a collection of tests to execute.

A scheme contains a set of targets (which may have dependencies among them), a configuration, and a set of tests to execute.

To illustrate the relationship among these five concepts with a perhaps imperfect analogy: an Xcode Workspace is like a factory, and an Xcode Project is like a workshop. Each workshop can produce products independently of the factory (a project can exist independently of a workspace), but when multiple workshops are combined, a factory is needed to organize them (if you use CocoaPods, you need to use a workspace). An Xcode Target is an assembly line, and a single assembly line produces only one type of product. Build Settings are the recipe for producing the product; if you are producing soda, Build Settings are the formula for the ingredients. An Xcode Scheme is the production plan: it includes the assembly line, the recipe, and quality inspection after production is complete (tests).


##### 2. Create an xcconfig file  

![](https://img.halfrost.com/Blog/ArticleImage/19_22.png)  

After creating this file, configure it in the project.

![](https://img.halfrost.com/Blog/ArticleImage/19_23.png)  


In these places, replace the configuration file with the file we just created.


Next, we need to write our xcconfig file. There are quite a few things you can put in this file. Careful readers will notice that CocoaPods, which we have been using all along, actually uses this file to configure build parameters. If we take a look at a simple CocoaPods xcconfig file, it looks like this:
```vim 

GCC_PREPROCESSOR_DEFINITIONS = $(inherited) COCOAPODS=1
HEADER_SEARCH_PATHS = $(inherited) "${PODS_ROOT}/Headers/Public" "${PODS_ROOT}/Headers/Public/Forms"
OTHER_CFLAGS = $(inherited) -isystem "${PODS_ROOT}/Headers/Public" -isystem "${PODS_ROOT}/Headers/Public/Forms"
OTHER_LDFLAGS = $(inherited) -ObjC -l"Forms"
PODS_ROOT = ${SRCROOT}/Pods
```
Since we need to configure the network environment, we can write it like this.
```vim 
//Network request baseurl
REQUESTBASE_URL = @"http:\\/\\/10.20.100.1"
```
Of course, it can also be written in the CocoaPods style.
```vim
GCC_PREPROCESSOR_DEFINITIONS = $(inherited) WEBSERVICE_URL='$(REQUESTBASE_URL)' MESSAGE_SYSTEM_URL='$(MESSAGE_SYSTEM_URL)'

```
This uses the GCC_PREPROCESSOR_DEFINITIONS build setting.

>Space-separated list of option specifications. Specifies preprocessor macros in the form foo (for a simple #define) or foo=1 (for a value definition). This list is passed to the compiler through the gcc -D option when compiling precompiled headers and implementation files.

GCC_PREPROCESSOR_DEFINITIONS is a GCC precompiled-header setting. Typically, we can assign default values for preprocessor macro definitions under Build Settings in the Project file.


It is located in Build Settings under Apple LLVM 7.X - Preprocessing - Preprocessor Macros.  

![](https://img.halfrost.com/Blog/ArticleImage/19_24.png)  


Preprocessor Macros are actually configured by default according to the Configuration option. They allow you to predefine different macros for different environments, or define different values for the same variable in different environments.

With xcconfig, we can write different Configuration options into different files. Each xcconfig can configure attribute values in Build Settings. In essence, this modifies the value of GCC_PREPROCESSOR_DEFINITIONS through xcconfig, which lets us meet the requirement of dynamic environment configuration.


One final point worth mentioning is the level of this configuration file. With so many local configurations, which one actually takes effect in the end? Open the levels in Build and let’s look at an example.

![](https://img.halfrost.com/Blog/ArticleImage/19_25.png)  


We can currently see five configurations, and they have priorities. The priority decreases from left to right. Resolved = target-level > project-level > custom configuration file > iOS default configuration. The first column on the left always shows the final configuration result currently in effect.

Once we understand this priority order, we can configure our app more flexibly.


That is basically all for the basic usage of xcconfig configuration. But there is much more to it than this.

You can also use xcconfig to dynamically configure many parameters in Build Settings. This is actually similar to how CocoaPods does it. But one expert’s approach is very elegant and worth learning for anyone interested. The open-source library [xcconfigs](https://github.com/jspahrsummers/xcconfigs) by iOS expert [Justin Spahr-Summers](https://github.com/jspahrsummers) provides a quasi-authoritative template. It is a very good library for learning how to use xcconfig, and I highly recommend it.

#### 3. Using Targets to Configure Multiple Environments

To configure multiple environments, a Scheme plus xcconfig is already completely sufficient. So why do we need this third point? Although using multiple Targets just to configure multiple environments is a bit of an overkill, it can indeed satisfy the requirement.


As for the technique of building with Targets, I practiced it at a company two years ago. The requirement at the time was to build an OEM product. Our company had its own main product, and also built OEM products for other companies. Once OEM is mentioned, everyone should understand the clever use of Targets here. With Targets, you can instantly generate a large number of apps in bulk.

In 2013, Qiao Ge also published an article about Targets: [Technical Details of the Yuantiku iOS Client (Part 1): Using Multiple Targets to Build Many Similar Apps](http://blog.devtang.com/2013/10/17/the-tech-detail-of-ape-client-1/). My former company also implemented this functionality in 2014.

![](https://img.halfrost.com/Blog/ArticleImage/19_26.png)  

With just one codebase, you can produce seven apps. The certificates for the seven apps are all different, and their configurations are also different, but the code only needs to be maintained in one place to achieve the goal of maintaining seven apps.

Next, let’s look at how to create new Targets. There are two methods.

![](https://img.halfrost.com/Blog/ArticleImage/19_27.png)  


One method is to create a completely new Target, and the other is to duplicate an existing Target.

In fact, after creating a Target with the first method, what you do next depends on your requirements. If you also want to build something like an OEM product, you can delete the newly created project and still maintain only one local codebase. Then, in the Build Phases of the newly created Target, add the existing local code and configure the parameters however you like. This also lets one codebase maintain multiple apps.

The second method is to duplicate an existing Target. With this approach, you only need to modify the parameters yourself.

Now let’s talk about Target parameters.

Since we have created a new Target, it is equivalent to creating a new app. Therefore, all files inside it can be changed, including info.plist, source code references, Build Settings... all parameters can be modified. This is no longer limited to modifying Scheme and xcconfig. That is why I said earlier that using Targets merely to configure multiple environments is somewhat overkill, but it can indeed achieve the goal. As we also mentioned in the second chapter, Targets are equivalent to pipelines and are second only to the Project in importance. You can imagine that with Targets, there is almost nothing we cannot modify.


P.S. One final point about Targets: if you have multiple apps and more than 80% of the code among these apps is exactly the same, or if only a few screens are displayed differently while the logic is completely the same, I recommend using Targets. This way, you only need to maintain one codebase. Maintaining multiple copies of the same code is really inefficient. A single bug has to be fixed back and forth across multiple codebases, which is time-consuming and labor-intensive.

At this point, someone might ask: if we maintain one codebase, what happens if these apps have different requirements later? For example, what if they need to enter different screens, navigate to different screens, or display different pages? This problem is actually very simple. In **Compile Sources** under Targets, you can add different source files to compile for each different Target. You only need to add the code for different screens to the corresponding Targets for compilation, and add macros to the screen that performs navigation to control which app navigates to which corresponding screen. This way, you still maintain one local codebase, but the code compiled by each Target is a subset of that codebase. This remains very convenient to maintain, while also supporting different screens and different requirements for different apps.
